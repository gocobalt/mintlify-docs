Databricks supports three authentication methods. Choose the one that best fits your use case:

- **API Key Authentication** — Best for internal integrations using a Personal Access Token
- **OAuth 2.0 (Authorization Code with PKCE)** — Best for end-user authorization; no client secret required
- **OAuth 2.0 Client Credentials (M2M)** — Best for unattended, service-to-service access using a Service Principal

## Authentication at a glance

Before you pick a method, here is how each behaves in production — how much effort a connection takes to set up and keep alive, and how long it stays authenticated before someone has to act. Lifetimes below reflect [Databricks' own token defaults](https://docs.databricks.com/aws/en/admin/access-control/tokens).

| Method | Underlying standard | Credentials Refold stores | User effort | Authentication lifetime & re-auth |
| --- | --- | --- | --- | --- |
| **API Key** | Personal Access Token (bearer) | PAT + Base URL | **High, recurring** — a user generates the token by hand and must enter a new one each time it expires | You set the lifetime when creating the token (Databricks default max **730 days / 2 years**; workspace admins can enforce shorter with a max-token-lifetime policy). Tokens **unused for 90+ days are auto-revoked**, and Databricks emails ~7 days before expiry. When a token expires or is revoked, the connection fails until a new one is supplied. |
| **OAuth 2.0 (Authorization Code + PKCE)** | OAuth 2.0 auth-code with PKCE (no client secret) | End-user access + refresh tokens; Client ID + Workspace URL at the app level | **Low** — the end user authorizes once in the connect flow; no secrets for them to manage | Access tokens last **1 hour**; Refold refreshes them automatically using single-use refresh tokens (each refresh rotates the token). The connection stays alive as long as it is used and access is not revoked. Re-auth is needed only if the user revokes access or the refresh token is invalidated. |
| **OAuth 2.0 Client Credentials (M2M)** | OAuth 2.0 client-credentials (Service Principal) | Client ID + Client Secret + Base URL | **Low, no end user** — an admin sets up the service principal once; unattended thereafter | Refold mints a fresh **1-hour** access token on demand from the client secret, with no human involvement. The **client secret** is valid for up to **2 years** (max 730 days, set when generated; up to 5 secrets per principal for rotation). Rotate or replace the secret before it expires, or the connection breaks. |

<Note>
**Rule of thumb.** For hands-off, long-lived service access, prefer **M2M** — no end user, no interactive refresh, and a secret that lasts up to two years. For access scoped to a specific end user, prefer **OAuth (PKCE)** — one consent, then automatic refresh. Use an **API Key** only for quick internal setups; it carries the most ongoing maintenance because tokens expire and must be replaced by hand.
</Note>

## API Key Authentication

To successfully connect with the **Databricks** integration using API Key, a Linked Account or an end-user will need to provide the following:

1. API Key (Personal Access Token)
2. Base URL

<Info>
  To understand how a Linked Account can get the above mentioned credentials, refer below.
</Info>

### Getting Credentials of Databricks

To acquire the required credentials and connect a Linked Account, please follow the steps mentioned below:

1. Log in to your Databricks workspace at your workspace URL (e.g., `https://<workspace-id>.azuredatabricks.net`).
2. Click on your profile icon in the top right corner and select `Settings`.

<img
  height="200"
  src="/images/Databricks/databricks_navigation.png"
  alt="Navigation for Personal Access Token"
/>

3. Navigate to `Developer` in the left menu > Click on the `Manage` button next to **Access tokens** as shown above.
4. Click `Generate new token`.

<img
  height="200"
  src="/images/Databricks/databricks_credentials.png"
  alt="Getting Personal Access Token"
/>

5. Provide a **Comment** (e.g., "Refold Integration") and set a **Lifetime** for the token. Select any particular scopes as should be accessible by the Refold Databricks connector. Click `Generate`.
6. Copy the token displayed on the screen. This is your **API Key**.

<img
  height="200"
  src="/images/Databricks/databricks_credentials_2.png"
  alt="Obtain the API Key"
/>

<Warning>
  The token will only be shown once. Copy and store it immediately. If lost, you will need to revoke and generate a new token.
</Warning>

7. Your **Base URL** is the workspace URL you used to log in (e.g., `https://<workspace-id>.azuredatabricks.net`).

The Linked Account or end-user now have all the credentials required to connect with Databricks using API Key authentication.

## OAuth 2.0 (Authorization Code with PKCE) Authentication

To setup your Databricks app in Refold for OAuth, you will need the following credentials from your Databricks account console:

- Client ID
- Workspace URL

<Info>
  Databricks OAuth uses PKCE (Proof Key for Code Exchange), so no client secret is required. You will need to register a new app connection in the Databricks account console.
</Info>

### Creating an OAuth App Connection in Databricks

To create a Databricks OAuth app connection and acquire the above mentioned credentials, please follow the steps mentioned below:

1. Log in to your Databricks account console at [https://accounts.cloud.databricks.com](https://accounts.cloud.databricks.com).

<img
  height="200"
  src="/images/Databricks/DB_landingpage.png"
  alt="Databricks Account Console Landing Page"
/>

2. Click on **Settings** in the left sidebar.
3. Navigate to the **App connections** tab.

<img
  height="200"
  src="/images/Databricks/db_appconnections.png"
  alt="App Connections in Databricks Settings"
/>

4. Click on **Add connection** in the top right corner.
5. Fill in the connection details:
   - **Application Name**: Give your app a descriptive name (e.g., "Refold-databricks-app").
   - **Redirect URLs**: Go to your [`Apps catalog`](https://app.refold.ai/applications) in **Refold** \> Search for `Databricks` \> `Settings` \> `Use your credentials` \> `Callback Url` \> Copy it and paste it here.
   - **Access scopes**: Select the required scopes (e.g., `SQL` or `All APIs`).
   - **Client secret**: Leave the **Generate a client secret** checkbox **unchecked** — Databricks OAuth with PKCE does not require a client secret.

<img
  height="200"
  src="/images/Databricks/db_addConnection.png"
  alt="Add Connection Form in Databricks"
/>

6. Click **Add**. A **Connection created** popup will appear displaying your **Client ID**. Copy it and store it securely.

<img
  height="200"
  src="/images/Databricks/db_clientId.png"
  alt="Client ID from Databricks App Connection"
/>

7. Navigate to **Workspaces** in the left sidebar, click on your workspace, and copy the **Workspace URL** shown in the configuration (e.g., `https://dbc-<id>.cloud.databricks.com`).

<img
  height="200"
  src="/images/Databricks/db_copyworkspaceurl.png"
  alt="Copy Workspace URL from Databricks"
/>

#### Configuring OAuth credentials in Refold

App settings page lets you configure the authentication settings for an `OAuth2` based application. For your customers to provide you authorization to access their data, they would first need to install your application. This page lets you set up your application credentials.

Provide the acquired **Client ID** and **Workspace URL** under Settings of the Databricks app in Refold and save it.

#### Configuring Scopes

Refold lets you configure what permissions to ask from your users while they install your application. The scopes can be added or removed from the App settings page, under `Permissions & Scopes` section.

For some applications Refold sets **mandatory scopes** which cannot be removed. Additional scopes can be selected from the drop down. Refold also has the provision to add any custom scopes supported by the respective platform.

<Warning>
  If you are facing scopes missing or invalid scope error, make sure the scopes selected in Refold match those configured in your Databricks app connection. And, ensure you are not passing any custom scope not supported by the platform.
</Warning>

## OAuth 2.0 Client Credentials (M2M) Authentication

Databricks supports Machine-to-Machine (M2M) authentication using a **Service Principal** with the OAuth 2.0 **Client Credentials** grant. Use it for unattended, service-to-service access where no end user is present to authorize.

To connect a Linked Account using M2M, you will need:

- Client ID
- Client Secret
- Workspace Base URL

<Info>
  Unlike the PKCE method, M2M uses a **client secret** and authenticates as a **Service Principal** rather than on behalf of an end user. Create the service principal in the Databricks **Account Console** as an **Account Admin**.
</Info>

### Getting M2M Credentials of Databricks

#### 1. Create a Service Principal

1. Log in to the **Databricks Account Console** as an **Account Admin**.
2. In the left sidebar, navigate to **User Management → Service Principals**.
3. Click **Add Service Principal**.
4. Enter a name for the service principal and click **Add**.

#### 2. Assign Roles

1. Open the newly created service principal.
2. Navigate to the **Roles** tab.
3. Assign the required account roles as appropriate for your use case.

#### 3. Grant Workspace Access

1. Navigate to **Workspaces** and select the workspace you want the service principal to access.
2. Open the **Permissions** tab and click **Add Permissions**.
3. Search for the service principal you created, assign the required permissions (for example, **User** or any other role your application needs), and save the changes.

#### 4. Generate OAuth Credentials

1. Navigate to **User Management → Service Principals** and select your service principal.
2. Open the **Credentials** tab and click **Generate Secret**.
3. Copy and securely store the **Client ID** and **Client Secret**.

<Warning>
  The client secret is displayed only once. Store it securely before closing the dialog. If lost, you will need to generate a new secret.
</Warning>

#### 5. Obtain the Workspace Base URL

Open your Databricks workspace. The workspace URL looks similar to `https://dbc-xxxxxxxx-xxxx.cloud.databricks.com`. Copy it — this is your **Base URL**.

#### Configuring M2M credentials in Refold

Enter the following values into the Databricks connection configuration in the hosted portal, then save:

| Field | Value |
| --- | --- |
| Base URL | Your Databricks workspace URL |
| Client ID | Generated Client ID |
| Client Secret | Generated Client Secret |

Once these values are provided, the connector authenticates using the **OAuth 2.0 Client Credentials (M2M)** flow.
