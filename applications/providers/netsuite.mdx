Refold connects to NetSuite through its **REST web services**. There are several ways to authenticate — pick one by weighing **how much setup effort** it takes against **how long the connection stays authenticated** before someone has to re-authorize.

## Authentication at a glance

| Method | Underlying standard | Credentials Refold stores | User effort | Authentication lifetime |
|---|---|---|---|---|
| **Key-Based (TBA)** | OAuth 1.0a (signed requests) | Account ID · Consumer Key/Secret · Token ID/Secret | **Medium** — create an integration record and an access token under a dedicated role | **No expiry** — tokens stay valid until revoked |
| **OAuth 2.0 — Authorization Code** | OAuth 2.0, 3-legged (confidential client) | Client ID/Secret · Account ID | **Lowest** — the user consents through the OAuth prompt | Re-auth each **Refresh Token Validity** window (default 168 h / 7 days, settable up to 30 days) |
| **Client Credentials (M2M)** | OAuth 2.0, 2-legged (JWT signed by a certificate) | Client ID · Certificate ID · private key | **Highest** — create an integration record **and** generate + map a certificate | Bounded by **certificate validity** (up to ~2 years), then rotate — no user re-auth |

<Info>
Rule of thumb: **lowest effort → OAuth Authorization Code** (but users re-authorize periodically); **longest unattended life → Key-Based or M2M** (more setup, little-to-no re-auth). The [in-depth comparison](#in-depth-token-lifetimes-and-re-authentication) below explains the token mechanics behind each.
</Info>

## In depth: token lifetimes and re-authentication

NetSuite's OAuth 2.0 grant types behave very differently around **token expiry and re-authentication** — the difference matters most for unattended, always-on integrations, where some methods run for years without human involvement while others force a periodic re-login.

### Method comparison

| | Key-Based (TBA) | OAuth 2.0 — Authorization Code | Client Credentials (M2M) |
|---|---|---|---|
| **Underlying standard** | OAuth 1.0a; every request HMAC-signed | OAuth 2.0, 3-legged (confidential client) | OAuth 2.0, 2-legged; JWT signed with a certificate |
| **A user must log in to authorize?** | No — an admin generates tokens once | **Yes** — an end user authorizes via redirect | No — machine-to-machine |
| **Credentials Refold stores** | Account ID, Consumer Key/Secret, Token ID/Secret | Client ID/Secret, Account ID + issued tokens | Client ID + Certificate ID + private key |
| **Access-token lifetime** | **Never expires** | ~60 minutes | ~60 minutes |
| **Refresh token** | None (not needed) | Yes | None |
| **Re-authentication interval** | Never (until the token is revoked) | Every **Refresh Token Validity** window — default **168 h (7 days)**, editable **1–720 h (30 days)**; not extended by workflow activity | Never — bounded by certificate validity |
| **How tokens renew** | No renewal — the token is static | Refold auto-refreshes access tokens; a fresh authorization is needed when the Refresh Token Validity window lapses | Refold auto-mints access tokens from the certificate; no user involved |
| **Manual re-authentication** | Only if the token is revoked, or the user/role is disabled | Every Refresh Token Validity window (raise toward 720 h to reduce) | **Never** while the certificate is valid |
| **Credential rotation** | Manual, at your discretion (tokens don't expire) | Rotate the client secret manually | **Rotate the certificate before it expires** |
| **Works with 2FA-required roles** | **No** | Yes | N/A (no user; runs under a machine role) |
| **Best for** | Existing unattended integrations (legacy) | User-attended connections acting as a specific user | New unattended, server-to-server integrations |

### What to watch out for

<AccordionGroup>
  <Accordion title="Key-Based / TBA — tokens never expire (a benefit and a liability)">
    - **No re-auth, ever** — because TBA tokens don't expire, the connection never needs a human to log back in. This is why it has been the default for unattended integrations.
    - **A leaked token stays valid until you revoke it.** There's no automatic expiry to contain the blast radius, so treat the Token ID/Secret as long-lived secrets and **rotate them deliberately** on a schedule.
    - **Tied to a user + role.** The token is bound to the NetSuite user and role it was created under. If that employee is deactivated or the role changes, the token breaks — use a dedicated integration user, not a person's login.
    - **Cannot be used with roles that require 2FA.**
  </Accordion>
  <Accordion title="OAuth 2.0 Authorization Code — re-authentication timing">
    Refold connects as a **confidential client** (Client ID + Client Secret). Its re-authentication interval is governed by **Refresh Token Validity** on the NetSuite integration record — **default 168 h (7 days)**, editable **up to 720 h (30 days)**. The refresh token expires at the end of that window and **isn't extended by workflow activity**, so the connection needs a fresh authorization each interval.

    - **Reduce how often users re-authenticate** by raising **Refresh Token Validity** toward its 720-hour (30-day) maximum. To avoid recurring logins entirely, use **Client Credentials (M2M)**, which has no refresh token and no periodic re-authentication.
    - **Re-authorizing is not the same as issuing new client credentials.** Recovering a lapsed connection just means the user re-runs the connect flow (log in + consent); Refold reuses the **same Client ID/Secret**. You only generate new credentials if you deliberately reset or rotate them in NetSuite.
  </Accordion>
  <Accordion title="OAuth 2.0 Client Credentials / M2M — no re-auth, but rotate the certificate">
    - **No refresh token, no recurring re-authentication.** After a one-time certificate setup, Refold mints a fresh access token from a signed JWT whenever it needs one — so day-to-day operation runs unattended, with no periodic user login. It's the OAuth 2.0 successor to TBA for machine integrations.
    - **The certificate is the thing that expires.** NetSuite client-credential certificates have an expiry, so the one manual task is **rotating the certificate before it lapses** — an expired certificate takes the integration down. Track the expiry date and rotate ahead of it.
    - Permissions come from the **machine role** mapped to the client, not from an end user.
    - Full setup — certificate generation and mapping — is on the [Client Credentials (M2M) page](/applications/providers/netsuite_m2m).
  </Accordion>
</AccordionGroup>

### FAQ

<AccordionGroup>
  <Accordion title="Why did my OAuth connection expire after ~7 days even though workflows were running?">
    Refold's OAuth is a **confidential** client, so the re-auth interval is set by **Refresh Token Validity** on the NetSuite integration record — the token expires at that window regardless of activity. It **defaults to 168 h (7 days)**. **Raise it toward 720 h (30 days)** to re-authenticate less often, or use **[Client Credentials (M2M)](/applications/providers/netsuite_m2m)** for no periodic re-auth.
  </Accordion>
  <Accordion title="Do I need to regenerate credentials when a connection expires?">
    No — just **re-authorize**: the user re-runs the connect flow and Refold gets fresh tokens using your existing Client ID/Secret. Only **generate new credentials** (and update them in Refold) if the credential *itself* changes — you reset the Consumer Key/Secret on the integration record, or rotate the M2M certificate.
  </Accordion>
</AccordionGroup>

<Warning>
**Plan for NetSuite's TBA deprecation.** As of NetSuite **2027.1**, *new* integrations can no longer be created with TBA for SOAP/REST web services and RESTlets (existing integrations keep working). For a new NetSuite connection, prefer **OAuth 2.0**, or **Client Credentials (M2M)** for unattended workloads.
</Warning>

<Info>
Whichever method you choose, Refold acts under the **role** the token or OAuth client runs as. That role's permissions and subsidiary restrictions determine exactly which records Refold can read and write — scope the role to least privilege.
</Info>

  <Tab title="Generic record operations">
  Beyond the curated set, Refold exposes the **full SuiteTalk REST record catalog** — the same standard operations across 150+ NetSuite record types (invoices, sales orders, purchase orders, journal entries, vendor bills, item fulfillments, subscriptions, and many more):

  - **Get List Of Records** — list records of a type.
  - **Get Record** — fetch a record by ID.
  - **Insert Record** — create a record.
  - **Insert Or Update Record** — upsert by external ID.
  - **Update Record** — update a record.
  - **Remove Record** — delete a record.
  - **Transform To…** — convert one record into a related one (e.g. Sales Order → Invoice / Item Fulfillment, Purchase Order → Vendor Bill / Item Receipt, Opportunity → Estimate).
  - **Execute SuiteQL** — run arbitrary SuiteQL against `/query/v1/suiteql`.

  <Tip>
  Use curated actions for the common records; drop to generic record operations (or **HTTP Request**) for any record type or endpoint not covered by a named action.
  </Tip>
