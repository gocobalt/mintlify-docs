---
title: "Workflows as MCP Tools"
description: "Expose tested workflows as single tool calls instead of letting agents orchestrate raw API sequences"
---

AI agents can call individual APIs, but they struggle with complex multi-step processes that need precise sequencing, error handling, and business logic. They skip steps, make inconsistent decisions, and have no built-in compliance checks.

The fix: instead of letting agents orchestrate complex processes, expose your proven workflows as single MCP tools. The agent calls one tool. The workflow handles the rest.

## Why this matters

When agents handle multi-step processes directly:
- They may skip steps or reorder them incorrectly
- The same process runs differently each time
- No approval workflows, compliance checks, or rollback on failure
- One failure breaks everything downstream

When you wrap the process in a workflow:
- Execution is deterministic. Same inputs, same result.
- Error handling, retries, and rollback are built in
- Business rules (approvals, compliance, validation) are enforced
- Every execution is audited

<Warning>
A single workflow failure in SAP can cascade across procurement, finance, and operations. Agents calling raw APIs have no guardrails against this. Workflows do.
</Warning>

## Example: SAP Procurement

An agent needs to handle "Buy a laptop from Dell for $1,200." This is an 8-step process in SAP.

<Tabs>
  <Tab title="Agent calling raw APIs">
    The agent has to figure out and execute all 8 steps:

    1. Validate vendor master data and compliance status
    2. Check budget availability and get approval
    3. Create purchase requisition with cost center
    4. Route through approval hierarchy
    5. Generate purchase order with release strategy
    6. Notify vendor with requirements
    7. Process goods receipt with three-way matching
    8. Handle exceptions (price variance, quality, delays)

    **Failure points:** The agent might skip compliance checks, miss budget approval, bypass the approval hierarchy, or fail to handle exceptions. Each of these creates real business risk.
  </Tab>

  <Tab title="Agent calling workflows">
    The same process, split into 3 workflows exposed as MCP tools:

    **Workflow 1: Vendor and Budget Validation**
    - Input: Vendor ID, material, quantity, expected price
    - Handles: vendor compliance, budget check, pre-approval
    - Built-in error paths: vendor not found, budget insufficient, compliance issues

    **Workflow 2: PO Creation and Routing**
    - Input: Validated vendor data, budget approval, delivery requirements
    - Handles: PR creation, approval routing, PO generation
    - Built-in error paths: approval rejection, budget changes, delivery conflicts

    **Workflow 3: Goods Receipt and Exceptions**
    - Input: PO number, delivery details, actual quantities and prices
    - Handles: three-way matching, invoice verification, quality inspection
    - Built-in error paths: price variance, quality issues, delivery delays

    The agent calls 3 tools in sequence. Each workflow handles its own complexity internally.
  </Tab>
</Tabs>

## When to use workflows vs raw actions

| Scenario | Use |
|----------|-----|
| Single API call (create contact, get invoice) | Raw action |
| Multi-step process with business logic | Workflow |
| Process requires approvals or compliance checks | Workflow |
| Order of operations matters | Workflow |
| Failure in one step should trigger rollback | Workflow |

## How to set it up

1. Build your workflow in the Refold workflow builder with proper error handling, retry logic, and validation
2. Test it thoroughly outside of MCP
3. Add it to your MCP server's **Applications** tab (workflows appear alongside actions). See [Server Configuration](/mcp/server-configuration#applications-tab).
4. The workflow shows up as a tool the agent can call. In direct mode it appears as `<app_slug>_<workflow_name>_workflow`; in agent mode it's reachable via `EXECUTE_ACTION` with `type: "workflow"`. See [Tools Reference](/mcp/tools-reference).

The agent sends input, the workflow runs to completion, and the result comes back. The agent never touches the intermediate steps. Each invocation is recorded in [MCP Logs](/mcp/logs) with the workflow ID, input payload, and final status.

<Info>
Workflows and raw actions can coexist on the same server. Use raw actions for simple operations and workflows for anything that needs sequencing or error handling.
</Info>
