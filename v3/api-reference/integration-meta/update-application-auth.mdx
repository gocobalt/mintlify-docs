---
title: Migrate Token
api: put /api/v2/public/migrate-token
description: Migrate authentication credentials (API keys, access tokens) to Refold for an application.
---

Migrate existing authentication credentials to Refold for a specific application and linked account. Use this endpoint when you have pre-existing tokens or API keys that you want to use with Refold integrations, or when programmatically setting up connections without the OAuth redirect flow.

<Info>
This endpoint creates a new integration connection if one doesn't exist, or updates the existing credentials if a connection is already present.
</Info>

## Authentication

<ParamField header="x-api-key" type="string" required>
  Your Refold API key. Find it in **Settings → Credentials**.
</ParamField>

<ParamField header="linked_account_id" type="string" required>
  The unique identifier of the linked account to migrate credentials for.
  
  **Example:** `user_12345`
</ParamField>

## Body Parameters

<ParamField body="slug" type="string" required>
  The application slug to migrate credentials for.
  
  **Examples:** `slack`, `salesforce`, `hubspot`
</ParamField>

<ParamField body="auth_object" type="object" required>
  The authentication credentials to migrate. The structure depends on the application's authentication type.
  
  <Warning>
    For OAuth2 applications, only `access_token` and `refresh_token` are permitted.
    For key-based applications, use the field names defined in the application's `auth_input_map`.
  </Warning>
  
  <Expandable title="OAuth2 Example">
```json
    {
      "access_token": "xoxb-123456789-abcdefghij",
      "refresh_token": "xoxr-987654321-zyxwvutsrq"
    }
```
  </Expandable>
  
  <Expandable title="Key-based Example (Slack)">
```json
    {
      "bot_token": "xoxb-your-bot-token",
      "workspace_token": "xoxp-your-workspace-token"
    }
```
  </Expandable>
  
  <Expandable title="Key-based Example (Custom)">
```json
    {
      "api_key": "your-api-key",
      "base_url": "https://api.example.com"
    }
```
  </Expandable>
</ParamField>

## Example: Migrate OAuth2 Tokens

Use this when you have existing OAuth tokens from a previous integration system:
```javascript
async function migrateOAuthTokens(linkedAccountId, slug, tokens, apiKey) {
  const response = await fetch(
    'https://api.gocobalt.io/api/v2/public/migrate-token',
    {
      method: 'PUT',
      headers: {
        'x-api-key': apiKey,
        'linked_account_id': linkedAccountId,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        slug: slug,
        auth_object: {
          access_token: tokens.accessToken,
          refresh_token: tokens.refreshToken
        }
      })
    }
  );

  return response.json();
}

// Usage
await migrateOAuthTokens(
  'user_123',
  'salesforce',
  {
    accessToken: 'existing_access_token',
    refreshToken: 'existing_refresh_token'
  },
  'your_api_key'
);
```

## Example: Migrate API Keys

Use this for key-based applications:
```javascript
async function migrateApiCredentials(linkedAccountId, slug, credentials, apiKey) {
  const response = await fetch(
    'https://api.gocobalt.io/api/v2/public/migrate-token',
    {
      method: 'PUT',
      headers: {
        'x-api-key': apiKey,
        'linked_account_id': linkedAccountId,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        slug: slug,
        auth_object: credentials
      })
    }
  );

  return response.json();
}

// Usage for Slack
await migrateApiCredentials(
  'user_123',
  'slack',
  {
    bot_token: 'xoxb-your-bot-token',
    workspace_token: 'xoxp-your-workspace-token'
  },
  'your_api_key'
);

// Usage for a custom key-based app
await migrateApiCredentials(
  'user_123',
  'copper',
  {
    api_token: 'your-copper-api-key',
    email: 'admin@company.com'
  },
  'your_api_key'
);
```

## Auth Object Requirements

| Auth Type | Permitted Fields | Notes |
|-----------|-----------------|-------|
| `oauth2` | `access_token`, `refresh_token` | Only these two fields are allowed |
| `keybased` | Fields from `auth_input_map` | Use the exact field names defined in the application |

<Warning>
Mixing OAuth2 and key-based fields in the same request will result in an error. Check the application's `auth_type` before making the request.
</Warning>
