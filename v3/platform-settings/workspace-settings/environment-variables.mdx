---
title: "Environment Variables"
description: "Store API keys, secrets, and configuration values outside your workflow logic using scoped environment variables."
---

Environment variables let you store values such as API keys, secrets, configuration data, or any other reusable keys outside of your workflow logic. Instead of hardcoding a value into a node, you define it once and reference it by name. The value is managed separately and injected at runtime, which keeps workflows clean, reusable, and secure.

<Note>
  Looking for event payloads, node outputs, or other values that flow through a run? Those are [data variables](/v3/concepts/data-variables), not environment variables. This page covers the named, scoped values you manage in settings.
</Note>

## Scopes

Every environment variable has a **scope** that determines where its value is set and where it can be used. The scope is pre-selected based on the level you are on and cannot be changed after creation.

| Scope | Where the value is set | Where it can be used | Reference syntax |
|---|---|---|---|
| **Workspace** | Workspace variables page in Settings | Within that specific workspace (Test, Production, or a custom workspace) | `{{env.workspace.variable-name}}` |
| **Linked Account** | Inside each [Linked Account's](/v3/concepts/linked-account) overview page | In workflows running under that Linked Account | `{{env.linked_account.variable-name}}` |
| **Workflow** | Inside the workflow's Environment Variables panel | Only within that specific workflow | `{{env.workflow.variable-name}}` |

## Create a variable definition

Create variable definitions from **Settings → Configuration → Environment Variables**.

<Steps>
  <Step title="Open the Environment Variables settings">
    Go to **Settings → Configuration → Environment Variables**.
  </Step>
  <Step title="Select a scope level">
    Choose **Workspace**, **Linked Account**, or **Workflow**. This sets the variable's scope, which you cannot change later.
  </Step>
  <Step title="Add the definition">
    Click **+ Add Definition** and fill in the fields described below.
  </Step>
  <Step title="Create it">
    Click **Create Definition**. The variable is registered at that scope and ready for a value to be assigned.
  </Step>
</Steps>

Each definition has the following fields:

| Field | Description |
|---|---|
| **Name** | The identifier used to reference this variable in your workflows. Use clear, descriptive names like `API_KEY` or `CLIENT_SECRET`. |
| **Description** | A short note on what this variable is used for. |
| **Scope** | Pre-filled based on the level you are on. Cannot be changed. |
| **Format** | The expected format of the value: `String`, `Number`, or `URL`. |
| **Encrypted** | Controls whether the variable's value is hidden in logs and the UI. See [Encrypted environment variables](#encrypted-environment-variables) below. |

## Set variable values

A definition registers the variable. You set its value separately, at the scope you chose.

<Tabs>
  <Tab title="Workspace">
    After you create a definition at the Workspace scope, it appears in the Workspace variables list in Settings. Click **+ Value** on the variable row to enter the value for that workspace.

    Each workspace (Test, Production, or a custom workspace) is separate, so you set the value independently in each. This is useful for variables like base URLs or API keys that differ between environments.
  </Tab>
  <Tab title="Linked Account">
    For Linked Account-scoped variables, go to **Linked Accounts** and open the specific account. In the right panel, find the **Environment Variables** section and click **+ Add**.

    In the dialog, select the variable definition and enter the value for that Linked Account. Once set, the variable is available to any workflow running under that account.
  </Tab>
  <Tab title="Workflow">
    For workflow-scoped variables, open the workflow and click the **+** tab at the top, then select **Environment Variables** from the menu.

    This opens the Environment Variables panel for that workflow. Click **+ Add Variable** to assign a value to a definition.
  </Tab>
</Tabs>

## Use a variable in a workflow

Once a variable has a value, reference it inside any node using the **Insert Variable** picker. Under the **Environment Variables** group, variables are listed by scope: WORKSPACE, LINKED ACCOUNT, and WORKFLOW. Selecting one from the picker inserts the correct syntax automatically.

```
{{env.workspace.base-url}}
{{env.linked_account.client-secret}}
{{env.workflow.private-key}}
```

## Encrypted environment variables

When a variable holds sensitive data such as an API key, client secret, or auth token, enable encryption on it. Encrypted variables are never exposed in run logs, execution history, or the UI, even when a node's request is inspected.

### Enable encryption

Toggle **Encrypted** to ON when creating the variable definition, before you click **Create Definition**. Once enabled, the variable appears in the list with a **Yes** badge in the Encrypted column.

### How the value is stored

After you set a value for an encrypted variable, you cannot read it back through the UI. Clicking the eye icon on the variable row shows a read-only popup with `[REDACTED]` in place of the actual value. This applies across all scope levels.

### How it appears in logs

When a workflow runs and uses an encrypted variable, the value is replaced with `[REDACTED]` in the request payload visible in the Run tab. The workflow still executes successfully: the actual value is passed securely at runtime, and only the display is masked.

### When to use encrypted variables

Enable encryption for any variable whose value should never appear in plain text in logs or the UI:

- API keys and auth tokens
- Client secrets and credentials
- Passwords or private URLs containing credentials
- Any end-user-specific value that should not be visible to other users or admins

<Note>
  Encrypted values are still passed to nodes at runtime. Encryption controls visibility in the UI and logs only; it does not affect whether the value is used during execution.
</Note>
