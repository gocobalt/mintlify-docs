title: "Rate limiting"
description: "Keep integrations within third-party and Refold rate limits using backoff, batching, and scheduling so calls succeed instead of getting throttled."
---

A **rate limit** is the cap a service places on how many requests it accepts in a window of time. When you move data through Refold, two sets of limits apply: the limits of the [connected app](/v3/developer/configure/connector/supported-apps-actions) you call, and Refold's own platform limits. Design for both so high-volume customers don't get throttled.

This page covers the patterns that keep you under those limits: backing off on rejection, batching work, and scheduling load. It assumes you already [read and write data](/v3/embedded/move-data/read-write) through actions or [workflows](/v3/developer/configure/workflows/overview).

## Where limits come from

Every call you make through Refold can be throttled at two layers:

- **The third-party app's limit.** Each app (Salesforce, HubSpot, NetSuite, and so on) enforces its own quota, often per account, per app, or per endpoint. This limit is set by the app, not by Refold, and varies by provider and by your customer's plan with that provider. When you exceed it, the app returns an error, commonly HTTP `429 Too Many Requests`.
- **Refold's platform limit.** Refold meters API and workflow execution traffic to keep the platform stable for every tenant. When you exceed it, Refold returns a throttling response that you handle the same way.

<Note>
A rate limit is shared across all the work you run for a given customer. A bulk backfill, a real-time sync, and ad-hoc reads all draw from the same quota with the connected app, so plan their combined volume per [linked account](/v3/developer/configure/linked-account), not per workflow.
</Note>

## Handle a throttled response

When a call is throttled, the request failed but the situation is temporary: the right move is to wait and retry, not to give up. Build retries into the layer where the call happens.

<Steps>
  <Step title="Detect the throttle">
    Treat a `429` (and provider-specific throttling codes) as retryable, separately from a `4xx` that signals a bad request you should not retry.
  </Step>
  <Step title="Wait before retrying">
    Pause before the next attempt. Respect a `Retry-After` header if the app sends one; otherwise back off on your own schedule.
  </Step>
  <Step title="Back off exponentially">
    Increase the wait after each failed attempt (for example 1s, 2s, 4s) rather than retrying immediately in a tight loop. Add a small random jitter so many retries don't realign and hit the limit together.
  </Step>
  <Step title="Cap the attempts">
    Stop after a fixed number of attempts and surface the failure instead of retrying forever. See [Error handling](/v3/embedded/best-practices/error-handling) for what to do when retries are exhausted.
  </Step>
</Steps>

<Tip>
Inside a [workflow](/v3/developer/configure/workflows/overview), wrap the call that can be throttled in a [Try & Catch node](/v3/developer/configure/workflows/nodes/try-catch-node) and set its **Retry Policy** to **Exponential** with a delay and a max-attempts cap. The platform retries the call for you and only runs the catch branch once the attempts are exhausted.
</Tip>

## Reduce how many calls you make

The most reliable way to stay under a limit is to make fewer, larger, better-timed requests.

- **Batch where the app supports it.** Many apps accept multiple records in one request. Sending 100 records in one batched call instead of 100 single calls cuts your request count by the same factor.
- **Page through reads deliberately.** Pull large result sets in pages and process each page before requesting the next, rather than fanning out many concurrent reads.
- **Sync incrementally, not fully.** A [scheduled or polling sync](/v3/developer/configure/workflows/overview) that fetches only records changed since the last run touches far fewer records than a full re-pull each time.
- **Cache what rarely changes.** Reference data such as field metadata or picklists can be fetched once and reused instead of re-fetched on every operation.
- **Deduplicate before you write.** Skip writes that would not change anything. See [Idempotency](/v3/embedded/best-practices/idempotency) for safe retry and dedupe patterns.

## Schedule load to smooth out spikes

A limit is about requests *per window*, so spreading the same work over time keeps you under it.

- **Stagger bulk jobs.** When onboarding a customer with a large backfill, run it in chunks over time instead of all at once.
- **Use the Delay node to pace loops.** Inside a [Loop node](/v3/developer/configure/workflows/nodes/loop), add a [Delay node](/v3/developer/configure/workflows/nodes/delay) between iterations so a batch of writes doesn't burst past the app's per-second cap.
- **Avoid synchronized schedules.** If many customers run the same [scheduled workflow](/v3/developer/configure/workflows/overview), an identical run time concentrates load. Spread start times where you can.

<Warning>
Retrying a throttled write without [idempotency](/v3/embedded/best-practices/idempotency) can duplicate data: the original request may have succeeded before the throttle response reached you. Make writes safe to retry before you add aggressive retry logic.
</Warning>

## Best practices

<Steps>
  <Step title="Treat throttling as expected, not exceptional">
    Build backoff and retry into every write path from the start, rather than adding it after a customer hits a limit in production.
  </Step>
  <Step title="Back off exponentially with jitter">
    Grow the wait between attempts and randomize it so retries spread out instead of stacking.
  </Step>
  <Step title="Batch and page instead of fanning out">
    Prefer one batched request over many small ones, and process reads page by page.
  </Step>
  <Step title="Sync incrementally">
    Move only what changed since the last run so steady-state volume stays low.
  </Step>
  <Step title="Make retries idempotent">
    Pair retry logic with [idempotency](/v3/embedded/best-practices/idempotency) so a retried call never duplicates an effect.
  </Step>
</Steps>
