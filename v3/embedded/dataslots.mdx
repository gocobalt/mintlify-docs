A **Dataslot** is a field type you add to a page to collect a value from your customer. In a connector's **Connect Portal → Fields**, the **Dataslots** tab (next to **Preview**) lists everything you can add: **Static Dataslots** that are always available, and **Dynamic Dataslots** that pull live data from the customer's connected account.

Adding a Dataslot to a page creates a [config field](/v3/embedded/config-fields) the customer fills in. This page covers every Dataslot type and the options you can set on each.

## Add a Dataslot to a page

<Steps>
  <Step title="Open the Fields section">
    Go to the connector's **Connect Portal → Fields**. The left pane lists your [Pages](/v3/embedded/config-fields#pages) and the fields on each.
  </Step>
  <Step title="Pick the page, then open Dataslots">
    Select the page where the field should live, then open the **Dataslots** tab on the right. The **Preview** tab beside it shows the page exactly as the customer will see it.
  </Step>
  <Step title="Select a Dataslot">
    Click a Dataslot to add it as a field on the chosen page.
  </Step>
  <Step title="Configure it">
    Edit the new field — set its [field settings](#field-settings) and any type-specific options — then save it for customer use.
  </Step>
</Steps>

## Field settings

Every Dataslot shares the same set of settings in the editor. Only the **Options** area changes by type (covered per type below).

| Setting | What it controls |
| --- | --- |
| **Name** | The field's label, shown to your customer. |
| **Help Text** | Optional guidance shown beneath the field. |
| **Required** | The customer must supply a value before they can save. |
| **Hidden** | Hides the field from the customer — for values you set programmatically. |
| **Allow Multiple** | Lets the customer pick more than one value. Available on **Select**, **Map v2** columns, and **Dynamic Dataslots**. |
| **Use Data Source** | Loads a **Select** or **Map v2** column's options live from a published workflow instead of a fixed list. |
| **Default Value** | The value applied when the customer leaves the field empty. Its input matches the type (a date picker for Date, a JSON editor for JSON, and so on). |
| **Cache** | Caches the resolved value so Refold doesn't refetch it on every run — useful for live options that rarely change. |
| **Dependency** | Makes this field depend on another field — pick a field, and this one's options or visibility update with the parent's value. |
| **Page** | The [Page](/v3/embedded/config-fields#pages) this field appears on. |

<Note>
**Dependency** powers cascading fields: a child field's options load only after its parent has a value — for example, pick a **pipeline**, then choose a **stage** within it. Pair it with a [Datasource](#use-a-datasource) so the child list is fetched using the parent's selection.
</Note>

## Dataslot types

| Dataslot | Collects | Type-specific options |
| --- | --- | --- |
| **Text** | A free-form string | — |
| **Number** | A numeric value | — |
| **Email** | A validated email address | — |
| **Date** | A calendar date | Date-picker default |
| **JSON** | A structured JSON value | JSON default editor |
| **Select** | One choice from a list | Static options or [Use Data Source](#use-a-datasource); Allow Multiple |
| **Map v2** | A mapping between two columns | LHS / RHS columns, each typed independently |
| **Rule Engine** | Conditional logic | LHS options + Conditional Code |

### Text, Number, Email, Date

These are plain inputs. They expose only the [field settings](#field-settings) above; the **Default Value** input matches the type (numeric for Number, a validated address for Email, a date picker for Date). Use them for simple values like a prefix, a threshold, a recipient, or a cutoff date.

### JSON

Collects a structured **JSON** value, edited in a JSON editor. Use it for a default payload or a list of values. Its **Default Value** is a JSON document.

### Select

A **Select** shows a list the customer chooses from. In **Options**, either **add a static list** you type yourself, or toggle **Use Data Source** to load options live from a published workflow (see [Use a Datasource](#use-a-datasource)). Enable **Allow Multiple** for a multi-select.

### Map v2

A **Map v2** Dataslot lets the customer line up two lists, so a value on one side maps to a value on the other — for example, matching the customer's HubSpot contact properties to the fields your integration expects. It has two columns, **LHS** and **RHS**, each configured independently:

- **Name** — the column heading.
- **Type** — **Select** (a static list, or **Use Data Source**) or a **Dynamic Dataslot** (e.g. *Map Contact Properties*, *List Pipeline Stages*) that pulls live values from the connected account.
- **Options** — for a Select column, the static list or **Use Data Source**, plus **Allow Multiple**.
- **Dependency** — make the column depend on another field.

The customer then maps each **LHS** value to an **RHS** value. Inside a workflow, the [Data Mapper node](/v3/developer/configure/workflows/nodes/data-mapper) applies the saved mapping to records.

The saved value is an object keyed by **LHS value → RHS value**, stored under the field's `id`. Read and write it with `config()` / `updateConfig()` — see [Manage configs from your code](/v3/embedded/config-fields#manage-configs-from-your-code).

### Rule Engine

A **Rule Engine** Dataslot collects conditional logic as **LHS / operator / RHS** rows — for example, "only sync deals where `amount` > `1000`." It exposes two extra areas:

- **Options** — the choices for the **LHS** column (a static list, or **Use Data Source**).
- **Conditional Code** — JavaScript that sets the **operator** and **RHS** columns dynamically based on the LHS selection, using `cobaltRuleEngine.setColumnProperties("operator", …)` and `cobaltRuleEngine.setColumnProperties("rhs", …)`, and pulling live values with `cobaltDataSource.getDataSourceResponse("<workflow>", {})`.

<Tip>
Reach for Rule Engine only when you need conditional rules whose operators or right-hand values change with the left-hand choice. For a simple fixed choice, a **Select** is clearer.
</Tip>

## Dynamic Dataslots

**Dynamic Dataslots** are slots the Refold team builds when a connector is deployed. They query the customer's connected account live — for HubSpot, things like **List Deal Pipelines**, **List Pipeline Stages**, **Map Contact Properties**, **List Company Properties**, or **Lifecycle Stages**. Because they're built per connector, they aren't always present; when the slot you need is listed, add it directly and it returns options specific to each customer's account. Their editor exposes the standard [field settings](#field-settings), including **Allow Multiple**.

## Use a Datasource

When the Dynamic Dataslot you need isn't available, supply the live data yourself. On a **Select** or a **Map v2** column, check **Use Data Source**, then under **Select Data Source** pick one of your published **Application Workflows**. Each linked account then pulls data straight from their own connected account — their pipelines, their custom properties.

### Build the datasource workflow

<Steps>
  <Step title="Create a workflow">
    In the connector, add a workflow and name it so it's easy to spot (for example, `List Contact Properties`).
  </Step>
  <Step title="Fetch the data">
    Add a node that retrieves the records you want to offer — a connector action, a [custom action](/v3/developer/configure/connector/custom-actions), or an [HTTP Request](/v3/developer/configure/workflows/nodes/http) node against the connected app.
  </Step>
  <Step title="Transform into name/value pairs">
    Use a [Transform](/v3/developer/configure/workflows/nodes/transform) or [Custom Code](/v3/developer/configure/workflows/nodes/custom-code) node to reshape the records into objects with exactly `name` (the label the customer sees) and `value` (the ID you use).

    ```javascript Transform query
    {
      "fields": $map(records, function($r) {
        { "name": $r.label, "value": $r.name }
      })
    }
    ```

    <Warning>
    Each option must be a `name`/`value` pair and nothing else, or the options won't render correctly.
    </Warning>
  </Step>
  <Step title="Return the options">
    End the workflow with a [Response](/v3/developer/configure/workflows/nodes/response) node containing the transformed list. Nothing connects after it.
  </Step>
  <Step title="Publish">
    **Publish** the workflow, then select it under **Use Data Source** on your Dataslot. An unpublished workflow won't appear in the list and won't run for customers.
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="A datasource-backed list shows no options">
    **Fix:** Confirm the customer's linked account is connected and the workflow behind the slot is **published** and ends in a Response node returning `name`/`value` pairs. Re-open the field after the connection completes.
  </Accordion>
  <Accordion title="Options render incorrectly">
    **Fix:** Each option must be an object with exactly `name` and `value` — no extra keys. Re-check the Transform query in your datasource workflow.
  </Accordion>
  <Accordion title="A dependent field's options never load">
    **Fix:** Check the child field's **Dependency** points at the right parent, and that the parent has a value. Cascading options resolve only once the parent is set.
  </Accordion>
  <Accordion title="The Dataslot I need isn't in the Dynamic Dataslots list">
    **Fix:** Dynamic Dataslots are added per connector during listing, so the one you want may not exist. Build it yourself with [Use a Datasource](#use-a-datasource).
  </Accordion>
</AccordionGroup>
