An MCP server has three configuration tabs in the Refold dashboard: **Overview**, **Applications**, and **Skills**. Use these tabs to control the agent's identity context, its access boundary, and the reusable procedures it can load.

## Overview tab

<Frame>
  <img src="/images/mcp-server-overview.png" alt="Server Overview tab showing name, description, Server URL, and access toggles" />
</Frame>

### Name and description

The **Name** and **Description** are surfaced to the agent when it connects. Write them for the agent, not for humans — be specific about which apps and actions are available so the agent knows what it can do.

### Server URL

```
https://<domain>/mcp/v1/<token>/<server_id>
```

| Segment | What it is |
|---------|------------|
| `token` | Auth token tied to a specific [linked account](/v3/platform/concepts/linked-account). Each end user gets a unique token and therefore a unique URL. |
| `server_id` | Identifies this MCP server configuration. |

The token authenticates every request. Refold validates it on arrival and resolves the organization, linked account, and environment for the session. All tool calls within that session run under those credentials.

One server configuration is reused across all your customers — each [linked account](/v3/platform/concepts/linked-account) gets its own `token`. For how Refold issues and stores per-user URLs in production, see [How it works](/v3/mcp/get-started/architecture). For token security, revocation, and per-user scoping, see [Authentication](/v3/mcp/connect/authentication).

#### Optional query parameters

| Parameter | What it does |
|-----------|-------------|
| `?provider=<provider>` | Identifies the MCP client connecting (for example, `claude` or `cursor`). Used for client-specific behavior and surfaces the client name in [MCP Logs](/v3/mcp/operate/mcp-logs). |

<Info>
The Server URL is generated per linked account and retrieved from the dashboard.
</Info>

### Access controls

Three toggles in the **Server Details** panel control which tools and capabilities the agent can reach. These are the primary governance controls for scoping agent access.

| Toggle | What it enables | Default |
|--------|----------------|---------|
| **Auth Tool** | Adds the `AUTH` tool. When an app's credentials are missing, the agent returns an auth link instead of failing silently. | Off |
| **Retrieve Skill** | Adds `SEARCH_INTEGRATION_GRAPH` and `LOAD_SKILL` tools. The agent can discover and load pre-built procedures from your integration knowledge base. See [Skills](/v3/mcp/build/skills). | Off |
| **Agent Mode** | Switches from per-action tools to `RESOLVE_ACTIONS` + `EXECUTE_ACTION`. The agent resolves intent from natural language, then executes. | Off |

With all toggles off (the default), the server runs in **direct mode**: one tool per action or workflow, tightest scope.

<Note>
The **Agent Mode** and **Retrieve Skill** toggles affect which tools the agent sees and how it invokes actions — they also have access-control implications. See [Authentication](/v3/mcp/connect/authentication) for the security model.
</Note>

<Tip>
Start with direct mode and a small set of actions. Enable Agent Mode when you need broader coverage or when the number of actions makes individual tools unwieldy.
</Tip>

For the full decision framework — when to use direct, agent-mode, or workflow patterns — see [Choose your pattern](/v3/mcp/get-started/choose-your-pattern).

### Other settings

| Setting | What it does |
|---------|-------------|
| **Prompt suggestions** | Sample prompts shown in the embedded chat widget. Guides end users toward valid operations. |
| **Client setup** | Quick-reference connection instructions for different MCP clients. |
| **Embed chat** | Iframe snippet and standalone Chat URL for embedding the agent in your application. |
| **Debug mode** | Shows full request and response payloads in the embedded chat. Enable during development; disable in production. |

## Applications tab

<Frame>
  <img src="/images/mcp-add-application.png" alt="Applications tab with an app added and actions selected" />
</Frame>

The **Applications** tab lists the apps connected to this server and defines the agent's access boundary. Only apps added here are reachable. Only the actions you select within each app are exposed as tools.

Each app must be:

1. **Configured** in your Refold account with valid auth settings — see the [connector setup guide](/v3/developer/guide/build/connectors/overview).
2. **Authorized** by the linked account — the end user must have completed the auth flow for that app.

You can connect multiple apps to a single server. A server with Salesforce, SAP, and NetSuite gives the agent access to CRM, ERP, and financial actions in one session.

<Warning>
Adding an app does not expose all of its actions. You select which actions to expose when adding the app. Only expose what agents actually need.
</Warning>

## Skills tab

<Frame>
  <img src="/images/mcp-skills-tab.png" alt="Skills tab showing a custom skill with stats and status" />
</Frame>

The **Skills** tab manages the reusable procedures attached to this server. Skills give the agent step-by-step guidance for complex multi-step operations instead of letting it determine the approach on its own.

Skills attached here are scoped to this MCP server only. See [Skills](/v3/mcp/build/skills) to learn how to create and manage them.
