---
title: "Attach Skills"
description: "Attach reusable, tested procedures to your MCP server so agents follow defined steps instead of improvising."
---

A skill is a markdown document attached to an MCP server: a reusable procedure (or reference) the agent loads as content. Use skills to give agents structured guidance for operations that span multiple API calls, such as "sync contacts from Salesforce to HubSpot."

Without a skill, an agent works out a multi-step operation on its own. With a skill, it follows a tested procedure with defined steps, pre-checks, and error handling. Each skill is scoped to the MCP server it's attached to.

## When to use a skill

A skill wraps logic; a tool wraps a single call. Start with [per-action tools](/v3/mcp/tools) and add a skill when you notice agents struggling with multi-step operations or making inconsistent decisions about step ordering. For the full direct-vs-agent-vs-workflow decision, see [Choose an access pattern](/v3/mcp/build/server-config#choose-an-access-pattern); for procedures you want Refold to run deterministically rather than the agent, see [Expose Workflows as Tools](/v3/mcp/build/workflows-as-tools).

| | Skills | Per-action tools |
|--|--------|------------------|
| **Scope** | Multi-step procedures with logic | Single API calls |
| **Agent guidance** | Agent follows your defined steps | Agent decides the approach |
| **Error handling** | Defined in the procedure | Agent handles on its own |
| **Best for** | Complex business processes | Simple CRUD operations |

## How agents discover and use skills

When **Retrieve Skill** is enabled (in either direct or agent mode), the server gains two tools: `GET_KNOWLEDGE_INDEX` and `LOAD_SKILL`. See [Tools](/v3/mcp/tools#skill-tools) for their full parameter reference.

Agents discover skills by browsing, not by having them pushed up front:

1. **`GET_KNOWLEDGE_INDEX`** returns a compact, server-scoped index of the skills available on this server. (The tool name says "knowledge"; the index it returns is this server's skills.) Each entry carries a name, an ID, and a short description. Page through it with `limit` (default 50) and `offset` (default 0).
2. **`LOAD_SKILL`** loads the full markdown content of one skill by ID, on demand, once the agent picks a match.

The agent never receives full skill content up front. It scans the index, finds the skill that fits the task, then loads only that one:

```text
User: "Qualify this lead from the webinar"

1. The agent calls GET_KNOWLEDGE_INDEX to list the skills on this
   server, and sees "Lead Qualification" in the index.

2. It calls LOAD_SKILL(skill_id="...") to load the full procedure.

3. With the procedure loaded, the agent follows the steps. In agent
   mode it calls RESOLVE_ACTIONS then EXECUTE_ACTION for each
   integration action the skill requires; in direct mode it calls the
   per-action tools directly.
```

## Create a skill

<Steps>
  <Step title="Open the Skills tab">
    In your MCP server, go to the **Skills** tab and click **+ Add Skill**.
    <Frame>
      <img src="/images/mcp-skill-create.png" alt="Creating a skill with name, summary, app, content, and status" />
    </Frame>
  </Step>
  <Step title="Fill in the skill fields">
    | Field | Description |
    |-------|-------------|
    | **Name** | What the agent sees during discovery. Be descriptive, such as "Salesforce Lead Qualification Process". |
    | **Summary** | A short description of what the skill does and when to use it. |
    | **App** | Which app the skill relates to. |
    | **Content** | The procedure body: steps, API patterns, conditional logic, input/output specs. |
    | **Status** | Enable or disable the skill without deleting it. |
  </Step>
  <Step title="Write the procedure">
    Write the content as structured instructions an agent can follow. Mix prose with the concrete API or query patterns the agent should use (SOQL queries, REST endpoints, parameter names) so it doesn't have to guess them at runtime. Structure each procedure around these sections:

    - **Goal** states, in one line, what the skill accomplishes.
    - **Pre-checks** are the conditions that must hold before the agent proceeds (required inputs, an active connection).
    - **Steps** are the ordered actions, including the exact action IDs and query patterns to call.
    - **On success** defines what to return when the procedure completes.
    - **On failure** defines what to do when a step fails, including whether to stop or retry.
  </Step>
</Steps>

<Tip>
Write a skill the way you'd write a runbook for a junior team member: clear steps, explicit conditions, no ambiguity.
</Tip>

Click any skill in the **Skills** tab to see its rendered content, metadata, and usage stats.

## Examples

The actions referenced below are real Refold action IDs. Copy the structure and adapt the field names to your data. To find action IDs for an app on your server, browse the **Applications** tab in the dashboard. Every action exposed to the agent shows its action ID.

<Note>
Most Refold connectors use `snake_case` action IDs (`create_contact`, `get_purchase_orders`). NetSuite uses `camelCase` (`createAccount`, `getVendorById`). Use the exact ID for each app.
</Note>

### Lead qualification (Salesforce)

A single-app skill that combines a SOQL lookup, a conditional create, and scoring.

```markdown
## Goal
Qualify an inbound lead and capture the result on a Salesforce contact.

## Pre-checks
- Email address is provided
- Salesforce connection is active

## Steps

1. Look up the contact in Salesforce.
   Use action `get_entity_with_soql` with:
   `SELECT Id, AccountId, OwnerId FROM Contact WHERE Email = :email LIMIT 1`

2. If a contact exists, check for open opportunities on its account:
   Action `get_entity_with_soql` with:
   `SELECT Id, StageName FROM Opportunity WHERE AccountId = :accountId AND IsClosed = false`

3. If no contact exists, call `create_contact` with:
   `{ FirstName, LastName, Email, Phone, additional_fields: { LeadSource: "Webinar" } }`

4. Score the lead yourself (no API call) based on company size:
   > 500 employees → "high"
   200-500 → "medium"
   < 200 → "low"

5. Persist the score by updating the contact: action `update_contact` with
   `{ Id: <contactId>, additional_fields: { Lead_Score__c: <score> } }`.

## On success
Return the contact ID, qualification score, and any open opportunity IDs.

## On failure
If Salesforce is unreachable, return an error asking the user to retry.
```

### Cross-app contact sync (Salesforce → HubSpot)

A two-app skill that reads from one system, writes to another, and updates existing records instead of creating duplicates.

```markdown
## Goal
Mirror a Salesforce contact into HubSpot. Update if it already exists.

## Pre-checks
- Salesforce contact ID is provided
- Both Salesforce and HubSpot are connected on this server

## Steps

1. Fetch the contact from Salesforce. Use action `get_entity_with_soql`:
   `SELECT Id, FirstName, LastName, Email, Phone, Account.Name FROM Contact WHERE Id = :sfdcContactId`

2. Look up the contact in HubSpot by email:
   Action `search_contacts` with the email filter.

3. If found, call `update_contact` with the HubSpot contact ID and the
   mapped fields from step 1.

4. If not found, call `create_contact` with the mapped fields.

5. Return both the Salesforce and HubSpot contact IDs in the result.

## Field mapping

| Salesforce   | HubSpot   |
|--------------|-----------|
| FirstName    | firstname |
| LastName     | lastname  |
| Email        | email     |
| Phone        | phone     |
| Account.Name | company   |

## On failure
- If Salesforce lookup fails: stop, return the error to the user.
- If HubSpot is unreachable: do not retry indefinitely. Return the error
  with the partial data fetched from Salesforce.
```

### Purchase order with supplier validation (SAP S/4HANA)

A skill with strict ordering and explicit failure paths. Use this pattern for ERP processes where a wrong step has downstream cost.

```markdown
## Goal
Create a purchase order in SAP S/4HANA after validating the supplier.

## Pre-checks
- Supplier business partner ID, material, quantity, and expected price are provided
- SAP S/4HANA connection is active

## Steps

1. Validate the supplier. Call `get_business_partner_by_id` with the provided
   BusinessPartner ID. Confirm the record exists and check the relevant role
   and blocking fields in the response.

2. (Optional) List recent purchase orders for the supplier to surface any
   in-flight POs the user should know about. Use action `get_all_purchase_orders`
   with a `$filter` like `Supplier eq '<id>' and PurchaseOrderType eq 'NB'`.

3. Create the PO via `create_purchase_order` with the header and item payload
   SAP S/4HANA requires (PurchaseOrderType, CompanyCode, PurchasingOrganization,
   items, and so on).

4. Return the new PurchaseOrder ID, supplier, and total value.

## On failure
- Supplier not found: stop at step 1. Return the error with the BusinessPartner
  ID the user provided.
- PO creation fails: do not retry. The user needs to see the error to decide
  whether to fix the input.
```

These three examples cover the common shapes: single-app lookup-and-act, cross-app sync, and strict business-logic enforcement on an ERP process. Copy the closest one and adapt the actions to your own apps.
