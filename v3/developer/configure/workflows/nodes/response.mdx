The Response Node serves as the final output handler for workflow executions. It stores and formats response data returned via the Workflow API, allowing you to define custom status codes and format output data for external consumption. Use this node to make workflow results accessible when calling workflows programmatically.

## When to Use

- Return structured data after workflow completion via Workflow API
- Format API responses with custom status codes for different outcomes
- Store workflow results for access via Workflow API calls with `sync_execution: true`
- Provide data for Config DataSource listings

<Info>The Response Node should typically be the last node in your workflow. All workflow branches should eventually connect to a Response Node to ensure proper API responses.</Info>

## Input Parameters

<ParamField path="Status Code" type="number" required>
  HTTP status code to return with the response. Common values: `200` (success), `201` (created), `400` (bad request), `500` (error).
</ParamField>

<ParamField path="Data" type="textarea" required>
  Response data to return from the workflow. Supports JSON objects and variables from previous nodes. Use templating syntax to include dynamic values like `{{node_name.body.field}}`.
</ParamField>

<Info>Status codes and response data are not visible during node-level testing in the workflow builder. They become available when testing the workflow using **Try API** in the dashboard or when executing the workflow through the Workflow API.</Info>

## Output

The Response Node stores formatted response data accessible through the Workflow API.

**When called with `sync_execution: true`:**
```json
{
  "return_value": {
    "vendor": {
      "id": "V001234",
      "name": "Acme Corporation",
      "sync_status": "completed"
    },
    "timestamp": "2024-01-15T14:30:00Z"
  },
  "execution_id": "66e404464f475f36c0c95b5d"
}
```

**Response Accessibility:**
| Method | Availability |
|--------|--------------|
| Workflow API with `sync_execution: true` | Response returned in `return_value` field |
| Workflow API with `sync_execution: false` | Use `/response` API with `execution_id` to fetch |
| Try API in Dashboard | Response visible in test results |
| Node-level testing | Status code and data not visible |

## Adding to Your Workflow

<Steps>
  <Step title="Add Response Node">
    In the workflow editor, click **Add Node** and select **Response** from the Utility Nodes section. Place it as the final step in your workflow.
  </Step>
  <Step title="Configure status code">
    Set the appropriate HTTP status code:
    - **200**: Successful completion
    - **201**: Resource created
    - **400**: Validation failure
    - **500**: Processing error
  </Step>
  <Step title="Configure response data">
    In the **Data** field, enter the JSON structure to return. Use templating to include values from previous nodes:
```json
    {
      "success": true,
      "contact_id": "{{salesforce_node.body.id}}",
      "message": "Contact created successfully"
    }
```
  </Step>
  <Step title="Test via Try API">
    Use the **Try API** section in the dashboard to test your workflow and verify the response structure. Node-level testing does not display response data.
  </Step>
</Steps>

## Data Sources

Response data can include variables from multiple sources:

| Source | Syntax | Example |
|--------|--------|---------|
| Node Output | `{{node_name.body.field}}` | `{{sap_vendor.body.NAME1}}` |
| Start Node Payload | `{{start.body.field}}` | `{{start.body.vendor_id}}` |
| Instance Metadata | `{{instance_meta_data.field}}` | `{{instance_meta_data.instance_id}}` |
| DataSlot Values | `{{dataslot.field}}` | `{{dataslot.customer_info}}` |
| Static Values | Direct value | `"success": true` |

## Troubleshooting

<AccordionGroup>
  <Accordion title="Response Data Not Returned">
    | Problem | Solution |
    |---------|----------|
    | Response not in API call | Ensure `sync_execution: true` is set in the Workflow API request header |
    | Empty return_value | Verify Response Node is connected and executed in the workflow path |
    | Testing in workflow builder | Use Try API or Workflow API to see response data. Node-level testing does not display responses. |
  </Accordion>

  <Accordion title="Variable Resolution Issues">
    | Problem | Solution |
    |---------|----------|
    | Variables showing as undefined | Verify referenced nodes completed successfully before Response Node |
    | Incorrect variable syntax | Use `{{node_name.body.field}}` format. Check node output structure in logs. |
    | Nested field access | Use dot notation: `{{node.body.data.nested.field}}` |
  </Accordion>

  <Accordion title="Multiple Response Nodes">
    | Problem | Solution |
    |---------|----------|
    | Unexpected response data | Use Rule Node to direct workflow to a single Response Node based on conditions |
    | Conflicting responses | Ensure only one Response Node executes per workflow path |
  </Accordion>
</AccordionGroup>

<Warning>When using Response Node with Sub Flows, the response data from the sub flow is returned to the parent workflow. Use `sync_execution: true` in the Sub Flows node configuration to access sub flow responses.</Warning>
