Use these recommendations as a checklist when you design, ship, and maintain workflows. They cover naming, error handling, performance, security, testing, and observability so your workflows stay predictable as your integration catalog grows.

<Note>
  This page collects guidance that applies across every workflow you build in Refold. For node-level details, see the [workflow overview](/v3/platform/concepts/workflows/overview) and the individual node reference pages.
</Note>

## Naming and organization

Consistent names make workflows searchable, reviewable, and safe to refactor. Apply these conventions before your catalog grows past a handful of flows.

<AccordionGroup>
  <Accordion title="Use verb-first workflow names">
    Name each workflow after the action it performs, for example `sync-contact-to-hubspot` or `create-jira-issue-from-form`. Avoid generic names like `workflow-1` or `new-flow`.
  </Accordion>
  <Accordion title="Prefix workflows by domain or trigger">
    Group related flows with a shared prefix such as `crm/`, `ticketing/`, or `webhook/`. Prefixes make filtering, permissioning, and auditing easier.
  </Accordion>
  <Accordion title="Give every node a descriptive label">
    Replace default node titles (`HTTP Request`, `Transform`) with what the node actually does in context, for example `Fetch contact by email` or `Map Salesforce lead to Refold schema`.
  </Accordion>
  <Accordion title="Name variables for their meaning, not their type">
    Prefer `customerEmail` over `string1`, and `invoiceTotalCents` over `number2`. Include the unit when it matters. See [variables](/v3/platform/concepts/data-variables) for the variable system.
  </Accordion>
  <Accordion title="Document the trigger contract in the workflow description">
    Record which fields the trigger payload must contain, and what shape downstream consumers expect on success. New maintainers can then change the flow without rereading every node.
  </Accordion>
  <Accordion title="Split large flows into sub-flows">
    When a workflow exceeds roughly a screen of nodes, extract repeated logic into a [sub-flow](/v3/platform/concepts/workflows/nodes/sub-flows). Sub-flows give you reuse, unit-testable boundaries, and shorter diffs in review.
  </Accordion>
  <Accordion title="Keep data references intentional">
    Use [data refs](/v3/platform/concepts/workflows/nodes/data-ref-node) to pass structured payloads between nodes instead of stringifying and re-parsing JSON. Refs preserve types and surface schema drift earlier.
  </Accordion>
</AccordionGroup>

## Error handling and retries

Workflows talk to third-party APIs that fail, rate-limit, and change without notice. Build for those failures from the start.

<Steps>
  <Step title="Decide what each failure mode should do">
    For every external call, pick one of: retry transparently, skip and continue, branch into a recovery path, or abort the run. Make the choice explicit in the node, not implicit in default behavior.
  </Step>
  <Step title="Wrap external calls in retry with exponential backoff">
    Use the built-in retry settings on HTTP and integration nodes. Start with three attempts and exponential backoff so transient errors do not page anyone.
  </Step>
  <Step title="Distinguish retryable from terminal errors">
    Retry on `429`, `5xx`, and network errors. Do not retry on `4xx` validation errors. Surface terminal errors with the original status code and response body in the workflow log.
  </Step>
  <Step title="Add an error branch for downstream cleanup">
    When a partial run leaves data in an inconsistent state (created in system A, failed in system B), add an explicit error branch that compensates, for example deletes the orphan record or marks it as needing review.
  </Step>
  <Step title="Use idempotency keys for create operations">
    Pass a stable idempotency key (typically the trigger event ID) so retries do not produce duplicate records in the destination system.
  </Step>
</Steps>

<Warning>
  Silent failure is the worst failure. If you catch an error and continue, log the error and (where supported) attach it to the workflow run so the operator can see what happened.
</Warning>

<AccordionGroup>
  <Accordion title="Validate inputs at the top of the flow">
    Reject malformed trigger payloads in the first node. Failing fast at the boundary is cheaper than failing midway through an integration call.
  </Accordion>
  <Accordion title="Set timeouts explicitly">
    Do not rely on platform defaults. Set timeouts that match what the upstream API actually does, and make them shorter for user-facing flows than for background sync.
  </Accordion>
  <Accordion title="Cap retry budgets">
    Total retry time should be bounded. A flow that retries for an hour blocks downstream events behind it.
  </Accordion>
  <Accordion title="Send alerts only on terminal failure">
    Notify on the final error after retries are exhausted. Alerting on every transient retry generates noise and trains operators to ignore alerts.
  </Accordion>
</AccordionGroup>

## Performance and concurrency

Workflows that run on every event need to stay fast even as event volume grows. These patterns keep latency and cost in check.

<AccordionGroup>
  <Accordion title="Batch where the API supports it">
    If the destination accepts bulk endpoints, build batches in the workflow instead of running one node per record. Batching reduces API quota usage and rate-limit pressure.
  </Accordion>
  <Accordion title="Parallelize independent branches">
    When two downstream calls do not depend on each other, run them as parallel branches rather than serial nodes. Total latency drops to the slower of the two instead of the sum.
  </Accordion>
  <Accordion title="Avoid N+1 lookups">
    If you find yourself fetching one related record per item in a list, pull the related records in a single query first, then join in a [transform](/v3/platform/concepts/workflows/nodes/transform) node.
  </Accordion>
  <Accordion title="Cache static reference data">
    Field mappings, picklist values, and lookup tables that change rarely belong in a data ref or sub-flow that fetches once per run, not per record.
  </Accordion>
  <Accordion title="Move heavy logic into custom code nodes">
    Long chains of transform and rule nodes are slower and harder to read than a single [custom code](/v3/platform/concepts/workflows/nodes/custom-code) node. Use custom code when the logic is non-trivial.
  </Accordion>
  <Accordion title="Stream large payloads instead of holding them in memory">
    For exports and bulk syncs, page through results and process each page, rather than loading the full result set into a single variable.
  </Accordion>
  <Accordion title="Respect upstream rate limits">
    Add throttling on workflows that call rate-limited APIs in tight loops. A burst that trips a rate limit usually takes longer overall than a paced run.
  </Accordion>
</AccordionGroup>

<Tip>
  Measure before you optimize. Use the workflow run timeline to find the slow node, then change one thing at a time.
</Tip>

## Security and secrets

Workflows often carry customer data between systems. Treat that data and the credentials guarding it with the same care as production code.

<Warning>
  Never paste API keys, tokens, or passwords directly into workflow nodes, variables, or [templating](/v3/developer/configure/workflows/nodes/templating) expressions. Use the connector's stored credentials or environment-scoped secret references.
</Warning>

<Steps>
  <Step title="Use connector-managed auth wherever possible">
    Configured connectors handle OAuth refresh, key rotation, and audit logging for you. Custom HTTP nodes should be the exception, not the default.
  </Step>
  <Step title="Scope secrets to the environment that needs them">
    Production secrets do not belong in development workflows. Keep separate secret stores per environment and reference them by name.
  </Step>
  <Step title="Mask sensitive fields in logs">
    Mark fields like `password`, `ssn`, and `card_number` as sensitive so they are redacted in workflow run logs and error messages.
  </Step>
  <Step title="Validate webhook signatures">
    For workflows triggered by inbound webhooks, verify the signature header in the first node and reject unsigned or invalid requests before any business logic runs.
  </Step>
  <Step title="Limit data minimization">
    Pass only the fields downstream nodes need. Stripping unused fields early reduces blast radius if a log is leaked or a node misroutes data.
  </Step>
</Steps>

<AccordionGroup>
  <Accordion title="Rotate credentials on a schedule">
    Rotate connector credentials on the cadence your security team requires. Workflows using connector-managed auth pick up the new credential automatically.
  </Accordion>
  <Accordion title="Restrict workflow edit permissions">
    Only users who understand the integration should be able to edit production workflows. Use role-based access on the workflow editor.
  </Accordion>
  <Accordion title="Audit external destinations">
    Periodically review where each workflow sends data. A flow originally built to email a notification can drift into a data export over time.
  </Accordion>
</AccordionGroup>

## Testing and deployment

Treat workflows as code. The editor's convenience does not remove the need for repeatable testing and controlled rollouts.

<Steps>
  <Step title="Build a test event for every trigger">
    Save a representative payload for each [trigger](/v3/platform/concepts/workflows/triggers/overview) so you can replay it during development. See [workflow testing](/v3/platform/concepts/workflows/testing) for the test runner.
  </Step>
  <Step title="Cover error paths, not just the happy path">
    Run tests with malformed input, missing fields, and simulated upstream failures. A workflow that only works on clean data fails the first time a real user sends something unexpected.
  </Step>
  <Step title="Stage changes in a non-production environment">
    Make and test workflow changes in a dev or staging environment first. Promote to production only after the test run succeeds end-to-end.
  </Step>
  <Step title="Use version control on exported workflow definitions">
    If your team exports workflows to JSON, commit them to source control. Diffs make code review meaningful and rollback fast.
  </Step>
  <Step title="Roll out gradually">
    For high-volume triggers, enable the new flow for a subset of tenants or events before global rollout. A bug caught at one percent of traffic is much cheaper than one caught at one hundred percent.
  </Step>
  <Step title="Document the rollback plan">
    Before deploying a workflow change, write down the single action that reverts it: disable the flow, switch the trigger, or restore the prior version.
  </Step>
</Steps>

<Tip>
  Keep a curated set of regression test events in your repo. Run them after every significant change to catch silent breakages in mappings and transforms.
</Tip>

## Observability

You cannot fix what you cannot see. Build observability into each workflow so failures surface fast and root cause is obvious.

<AccordionGroup>
  <Accordion title="Log the trigger event ID at the top of every flow">
    A consistent correlation ID makes it trivial to follow one request across nodes, sub-flows, and external systems.
  </Accordion>
  <Accordion title="Log the inputs and outputs of every external call">
    Capture the request URL, status code, and a redacted view of the response body. When a third-party API misbehaves, this log is the only ground truth you have.
  </Accordion>
  <Accordion title="Emit business-level metrics, not just technical ones">
    Track records synced, contacts created, and tickets routed in addition to HTTP status codes. Business metrics catch logic bugs that technically succeed but produce the wrong result.
  </Accordion>
  <Accordion title="Set alerts on terminal failure rates">
    Alert when the rolling failure rate exceeds a threshold, not on every individual error. Threshold alerts catch real degradations without paging on a single bad payload.
  </Accordion>
  <Accordion title="Track sub-flow latency separately">
    A sub-flow that gets slower drags every parent flow down with it. Watch sub-flow timings independently.
  </Accordion>
  <Accordion title="Keep run history long enough to investigate">
    Set retention so you can replay and inspect at least the last few weeks of runs. Many bugs are only reported days after the fact.
  </Accordion>
</AccordionGroup>

## Maintenance habits

These habits keep the catalog healthy over the long run, separate from any single flow.

<AccordionGroup>
  <Accordion title="Review unused workflows quarterly">
    Disable or delete workflows that have not run in the last quarter. Stale flows accumulate stale credentials and confuse new maintainers.
  </Accordion>
  <Accordion title="Update integration provider configs when APIs change">
    When a provider deprecates an endpoint, update the workflow before the deprecation date. Reactive fixes are always more disruptive than scheduled ones.
  </Accordion>
  <Accordion title="Keep templates current">
    If your team uses workflow templates, update them when patterns change so new flows start from current best practice rather than last year's.
  </Accordion>
  <Accordion title="Record known limits">
    Document the per-tenant, per-event, and per-day limits that apply to each flow so on-call engineers can diagnose throttling without reverse-engineering the system.
  </Accordion>
</AccordionGroup>
