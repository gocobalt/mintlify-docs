Custom Output lets you declare the shape of a node's output and mark individual fields as encrypted. Declaring the shape makes those fields available in the downstream data picker so later nodes can reference them by name. Marking a field as encrypted stores its value securely and renders it as `[REDACTED]` in the Run tab, execution logs, and history, while still passing the real value to downstream nodes at runtime.

Use this when a node returns dynamic JSON (for example, a [Custom Code](/v3/developer/configure/workflows/nodes/custom-code) or [HTTP](/v3/developer/configure/workflows/nodes/http) node) and you want downstream nodes to consume typed fields, keep sensitive values out of logs, or both.

## Define a custom output schema

Custom Output sits on the node's **Fields** tab. Each entry is a key written as a dotted path relative to the node's `body` object.

### Field path syntax

All node outputs are nested inside a top-level `body` object. When you add a key in Custom Output, the path is relative to `body`, so omit the `body` prefix.

Given this node output:

```json
{
  "body": {
    "result": [
      { "id": 1, "skill": "iPaaS" },
      { "id": 2, "skill": "Platform" }
    ],
    "welcome": "Hello!"
  }
}
```

Enter these keys in the Custom Output section:

| Key | What it targets |
| --- | --- |
| `result.*.skill` | The `skill` field across every item in the `result` array. |
| `welcome` | The top-level `welcome` field inside `body`. |

The `*` wildcard traverses every item in an array. One `result.*.skill` entry covers every object in the list, so you do not need a separate entry per index.

## Mark a field as encrypted

Each key listed in Custom Output has an **Encrypted** toggle on the right. Encryption is set per field, so visible and redacted outputs can coexist in the same node.

| Toggle | Behaviour |
| --- | --- |
| Off | Value is visible in run outputs and logs. |
| On | Value is stored securely and shown as `[REDACTED]` in the Run tab and logs. |

<Steps>
  <Step title="Locate the field">
    In the Custom Output section, find the row for the output key you want to protect.
  </Step>
  <Step title="Flip the Encrypted toggle on">
    Toggle **Encrypted** on for that row. Repeat for any other sensitive fields.
  </Step>
  <Step title="Save and test">
    Save the node and run the workflow. Encrypted fields render as `[REDACTED]` in the Output panel of the Run tab while non-encrypted fields render their real values.
  </Step>
</Steps>

<Note>
  Encrypted output fields control visibility only. The real value is still passed in full to downstream nodes at runtime. Encryption does not block, alter, or strip the data in transit, so a later node can still read, transform, or send the value as needed.
</Note>

## Example

Take a Custom Code node that returns an auth token alongside a public greeting:

```json
{
  "body": {
    "result": [
      { "id": 1, "skill": "iPaaS" },
      { "id": 2, "skill": "Platform" }
    ],
    "welcome": "Hello!"
  }
}
```

Configure Custom Output with two keys:

- `result.*.skill` with **Encrypted** on
- `welcome` with **Encrypted** off

When the workflow runs, the Output panel shows:

```json
{
  "body": {
    "result": [
      { "id": 1, "skill": "[REDACTED]" },
      { "id": 2, "skill": "[REDACTED]" }
    ],
    "welcome": "Hello!"
  }
}
```

A downstream HTTP node referencing `result.*.skill` still receives the real `iPaaS` and `Platform` values at runtime. Only the UI, Run tab, and logs see `[REDACTED]`.

## When to use encrypted output fields

Use encryption for any field whose value must reach a downstream node but should never appear in plain text in the UI or execution history:

- API keys or auth tokens returned by a Custom Code node.
- User-specific data such as personal identifiers or contact details.
- Credentials or secrets fetched from an external API during workflow execution.
- Any field required downstream but that must not appear in logs in plain text.

## Limits and caveats

- Paths are written relative to `body`. Including the literal `body` prefix in the key will not match the output.
- The `*` wildcard applies to array traversal. It captures the named field on every item in the array at that level.
- Encryption is per field, not per node. You must toggle each sensitive field on individually.
- Encryption hides values from the UI, Run tab, and execution history only. Downstream nodes always receive the real value.
- Values are described as "stored securely." Specific algorithm, key management, and rotation details are not exposed in the product.
