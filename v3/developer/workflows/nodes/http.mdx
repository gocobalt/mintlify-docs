The HTTP Node enables you to make HTTP requests to external REST APIs and webhook endpoints during workflow execution. It supports all standard HTTP methods, custom headers, query parameters, request bodies, and built-in pagination handling for APIs that return paginated responses.

## When to Use

- Call external APIs to fetch or send data during workflow execution
- Integrate with third-party services that expose REST endpoints
- Send webhook notifications to external systems
- Retrieve additional data from microservices based on workflow context
- Interact with enterprise systems (SAP, NetSuite, Tipalti) via their REST APIs

<Info>For file transfers over SFTP or large file operations, use the [File Handler Node](/v3/developer/workflows/nodes/file-handler) instead.</Info>

## Input Parameters

### Request Configuration

<ParamField path="Method" type="dropdown" required>
  HTTP method for the request. Options: `GET`, `POST`, `PUT`, `DELETE`, `PATCH`.
</ParamField>

<ParamField path="Base URL" type="string" required>
  The base URL of the API endpoint. Example: `https://api.example.com`
</ParamField>

<ParamField path="Path" type="string" required>
  The API path appended to the base URL. Supports dynamic variables. Example: `/v1/users` or `/vendors/{{vendor_id}}`
</ParamField>

### Query Params

Add query parameters as key-value pairs. Click the **+** button to add additional parameters.

| Field | Description |
|-------|-------------|
| Name | Parameter name. Example: `limit`, `page`, `filter` |
| Value | Parameter value. Supports dynamic variables. Example: `10`, `{{page_number}}` |

### Headers

Add HTTP headers as key-value pairs. Click the **+** button to add additional headers.

| Field | Description |
|-------|-------------|
| Name | Header name. Example: `Authorization`, `Content-Type`, `X-API-Key` |
| Value | Header value. Supports dynamic variables. Example: `Bearer {{api_token}}`, `application/json` |

### Body

<ParamField path="Body" type="code editor">
  Request body for POST, PUT, and PATCH requests. Enter JSON, XML, or other formats as required by the target API. Supports dynamic variables using `{{variable}}` syntax.
</ParamField>

<Info>The Body field only appears when the method is set to POST, PUT, or PATCH.</Info>

### Settings

<ParamField path="Use mTLS" type="boolean">
  Enable mutual TLS authentication for secure API connections. When enabled, select the certificate source.
</ParamField>

<ParamField path="Certificate Source" type="toggle">
  Choose between `truststore` or `certificate` as the source for mTLS credentials.
</ParamField>

<ParamField path="Select Trust Store / Select Certificate Store" type="dropdown">
  Select the configured trust store or certificate store from your organization settings. Configure stores under **Settings > Organization > Certificates**.
</ParamField>

<ParamField path="Use Server Trust Store" type="boolean">
  Use the server's trust store for SSL certificate validation.
</ParamField>

### Pagination

<ParamField path="type" type="dropdown">
  Select pagination handling method for APIs that return paginated responses. Options: `No pagination`, `Page Based`, `Offset Based`, `Cursor Based`, `Time Based`, `SQL Query Based`, `Custom`.
</ParamField>

<AccordionGroup>
  <Accordion title="Pagination Types">
    | Type | Description |
    |------|-------------|
    | No pagination | Single request, no automatic pagination |
    | Page Based | Uses page number parameter to fetch subsequent pages |
    | Offset Based | Uses offset and limit parameters for pagination |
    | Cursor Based | Uses cursor tokens returned by the API for next page |
    | Time Based | Paginates based on timestamp ranges |
    | SQL Query Based | Uses SQL-style LIMIT and OFFSET for pagination |
    | Custom | Define custom pagination logic |
  </Accordion>
</AccordionGroup>

## Output

The node returns the complete API response. The output structure depends on the target API's response format.

**Success Response Example:**
```json
{
  "status": 200,
  "headers": {
    "content-type": "application/json",
    "x-request-id": "abc123"
  },
  "body": {
    "id": "12345",
    "name": "Acme Corporation",
    "status": "active"
  }
}
```

**Error Response Example:**
```json
{
  "status": "Errored",
  "node_id": "16",
  "node_name": "Send GET request",
  "body": {
    "message": "Invalid URL",
    "error": {
      "code": "ERR_INVALID_URL",
      "input": "//jsonplaceholder.typicode.com/todos/1"
    },
    "status": 500
  }
}
```

## Adding to Your Workflow

<Steps>
  <Step title="Add the HTTP Node">
    In the workflow editor, click **Add Node** and select **HTTP Node** from the Utility Nodes section.
  </Step>
  
  <Step title="Select HTTP method">
    Choose the appropriate method from the dropdown:
    - **GET** for retrieving data
    - **POST** for creating resources
    - **PUT/PATCH** for updating resources
    - **DELETE** for removing resources
  </Step>
  
  <Step title="Configure URL">
    Enter the base URL and path for your API endpoint. Use dynamic variables to include data from previous nodes.
  </Step>
  
  <Step title="Add headers and parameters">
    Switch between the **Query Params** and **Headers** sub-tabs to configure:
    - Authentication headers (Bearer tokens, API keys)
    - Content-Type headers
    - Query parameters for filtering or pagination
  </Step>
  
  <Step title="Configure body (if applicable)">
    For POST, PUT, or PATCH requests, enter the request body in JSON format. Reference data from previous nodes using `{{node_name.field}}` syntax.
  </Step>
  
  <Step title="Test the node">
    Click the **Run** tab to execute a test request and verify the response structure.
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Connection Errors">
    | Problem | Solution |
    |---------|----------|
    | Invalid URL error | Verify the base URL includes the protocol (`https://`). Check for typos in the URL or path. |
    | Connection timeout | Verify the target API is accessible. Check network connectivity and firewall rules. |
    | DNS resolution failed | Verify the hostname is correct. Check if the API endpoint is publicly accessible. |
  </Accordion>

  <Accordion title="Authentication Errors">
    | Problem | Solution |
    |---------|----------|
    | 401 Unauthorized | Verify API credentials are correct and not expired. Check the Authorization header format. |
    | 403 Forbidden | Confirm the API key or token has permissions for the requested resource. |
    | Invalid token format | Ensure Bearer tokens include the `Bearer ` prefix. Verify base64 encoding for Basic auth. |
  </Accordion>

  <Accordion title="Request Errors">
    | Problem | Solution |
    |---------|----------|
    | 400 Bad Request | Verify request body format matches API requirements. Check Content-Type header. |
    | 404 Not Found | Verify the API path is correct. Check if the resource exists. |
    | 405 Method Not Allowed | Verify the HTTP method is supported by the endpoint. |
  </Accordion>

  <Accordion title="mTLS Errors">
    | Problem | Solution |
    |---------|----------|
    | Certificate not found | Verify the trust store or certificate is configured under **Settings > Organization > Certificates**. |
    | SSL handshake failed | Check certificate validity and expiration. Ensure the certificate matches the API requirements. |
  </Accordion>
</AccordionGroup>
