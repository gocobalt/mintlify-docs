---
title: "Authentication"
description: "Configure how a connector authenticates, so your customers can securely connect their account — with OAuth 2.0 or a key-based method."
---

A connector's **authentication settings** define how your customers grant access to their account. You configure this once on the connector, in **App Settings**; each customer then authorizes through the [auth flow](/v3/native/frontend/overview) you ship, and Refold stores their credentials and signs every request on their behalf.

Authentication is set per connector because every provider is different — some use OAuth 2.0, others a static API key.

## Choose an authentication method

Pick the method the provider's API supports:

<Tabs>
  <Tab title="OAuth 2.0">
    The customer is redirected to the provider to authorize your app, and the provider returns tokens Refold stores and refreshes. Best when the provider supports it — the customer never handles a raw key.
  </Tab>
  <Tab title="Key-based">
    The customer enters credentials they create in the provider — an **API key**, or a set like account / consumer / token — directly in the hosted portal when they connect. Refold injects them into every request.
  </Tab>
</Tabs>

## Set up OAuth 2.0

OAuth is configured on the connector's **Settings → Credentials**, where you add the OAuth client credentials issued by the provider.

<Steps>
  <Step title="Add the client credentials">
    Paste the **Client Id** and **Client Secret** from the OAuth app you created in the provider's developer console.
  </Step>
  <Step title="Register the Callback URL">
    Copy the **Callback Url** shown here and add it as the redirect URL on the provider's OAuth app — they must match exactly. If you'll receive events from the provider, also register the **Single Webhook Url** shown alongside it.
  </Step>
  <Step title="Request scopes">
    Under **Permissions & Scopes**, choose the permissions to request — see [Scopes & permissions](#scopes-and-permissions).
  </Step>
</Steps>

<Frame>
  <img src="/images/app-settings/oauth-credentials.png" alt="The connector's OAuth credentials: Client Id, Client Secret, Callback Url, Single Webhook Url, and Permissions & Scopes" />
</Frame>

<Note>
Building a custom app? Its OAuth setup also asks for the **Authorization URL** and **Token URL**, and lets you map the provider's token response (access token, refresh token, expiry). For a listed connector, Refold preconfigures these.
</Note>

## Set up key-based auth

A key-based connector has **nothing to configure in App Settings** — Refold defines the credential fields it needs. Your customer enters their credentials **directly in the hosted portal** when they connect. For NetSuite, that's the **Account Id**, **Consumer Key / Secret**, and **Token Id / Secret**:

<Frame>
  <img src="/images/app-settings/connect-credentials.png" alt="The hosted portal connect screen for NetSuite, offering OAuth 2.0 and Key Based authentication with the key-based credential fields" />
</Frame>

<Note>
Building a custom app with key-based auth? There you define the scheme — **API Key**, **Bearer token**, or **Basic** — and the fields the customer provides.
</Note>

## Scopes and permissions

**Scopes** are the permissions your app requests when a customer authorizes it — they decide what data and actions it can access. The available scopes **depend on the provider**: each app exposes its own set.

- **Mandatory scopes** — always requested and can't be removed (for HubSpot, `crm.objects.owners.read` and `oauth`).
- **Add from the dropdown** — pick any scope the provider lists.
- **Add your own** — if the scope you need isn't listed, type it into the same field to add it.

<Frame>
  <img src="/images/app-settings/scopes.png" alt="The HubSpot Permissions & Scopes field with the scope dropdown open and the mandatory scopes listed below" />
</Frame>

<Warning>
The scopes you select here must **match the scopes configured on the provider's OAuth app**. A mismatch — or a custom scope the provider doesn't support — causes a "missing scope" or "invalid scope" error at authorization.
</Warning>

If you rely on [token refresh](/v3/platform/authentication/managing-connections/token-refresh), request the provider's offline-access / refresh-token scope so the provider returns a refresh token.

## Multiple auth methods

Some providers support more than one way to authenticate (for example, OAuth 2.0 **or** key-based, like NetSuite above). When a connector offers more than one, the customer picks the method they prefer in the hosted portal at connect time.

## Troubleshooting

<AccordionGroup>
  <Accordion title="“Missing scope” or “invalid scope” error">
    **Fix:** Make sure the scopes selected here are identical to those configured on the provider's OAuth app, and that you haven't added a custom scope the provider doesn't support.
  </Accordion>
  <Accordion title="Redirect / callback URL mismatch">
    **Fix:** The Callback URL registered in the provider app must match the one on the App Settings page exactly — including scheme and trailing path.
  </Accordion>
  <Accordion title="Connections work but stop after a while">
    **Fix:** The access token expired without a refresh token. Request the provider's offline-access / refresh-token scope and confirm the **refresh token** is mapped in the token response. See [Token refresh](/v3/platform/authentication/managing-connections/token-refresh).
  </Accordion>
  <Accordion title="Credentials rejected on connect">
    **Fix:** For key-based auth, confirm the customer entered every credential field correctly in the hosted portal and that the credentials are still valid in the provider. For OAuth, re-check the Client Id / Secret.
  </Accordion>
</AccordionGroup>
