**Refold Managed** — also called **Bring Your Own Cloud (BYOC)** — runs the
Refold platform inside your own AWS account. Refold provisions all
infrastructure on your behalf and operates it through the
[Control Plane](/v3/deployment/kubernetes/control-plane), while you keep full
ownership of your data and environment.

<Info>
  **Which Kubernetes option?**
  - [**Helm**](/v3/deployment/kubernetes/helm) — self-host and operate everything yourself in a connected cluster.
  - **Refold Managed (BYOC)** *(this page)* — Refold provisions and manages everything in your AWS account; pick this for data residency without operating the platform yourself.
</Info>

## Overview

How a BYOC deployment comes together:

<Steps>
  <Step title="Create an IAM role">
    Create a dedicated IAM role in your AWS account using the policies Refold
    provides.
  </Step>
  <Step title="Complete the onboarding form">
    Share your AWS details (account ID, region, role ARN, technical contact).
  </Step>
  <Step title="Refold provisions and deploys">
    Refold assumes the role, provisions all infrastructure, and deploys the
    agreed Refold version.
  </Step>
  <Step title="Manage upgrades yourself">
    Your DevOps team triggers future upgrades independently via the Refold
    control plane.
  </Step>
</Steps>

## Prerequisites

Before provisioning begins, create one IAM role in your AWS account and submit
the onboarding form.

### IAM role

Create a role named `refold-<client-name>-deployment-role`. Refold assumes this
role to provision and manage all infrastructure; your account manager provides
the trust principal (Refold's AWS account ARN).

| Policy | Purpose |
|---|---|
| **Managed policy** | Grants the permissions needed to provision infrastructure and deploy Refold. |
| **Inline (trust) policy** | Establishes the trust relationship so Refold can securely assume the deployment role. |

Your account manager shares both policy documents.

## What Refold provisions

After you submit the onboarding form, Refold assumes your role and builds the
following inside your AWS account:

| Component | Details |
|---|---|
| VPC & networking | Dedicated VPC, public + private subnets across multiple AZs, NAT gateway |
| EKS cluster | Managed Kubernetes with a private API endpoint; control plane managed by AWS |
| Node groups | Auto-scaling workload nodes — 8 vCPU, 32 GB RAM each (e.g. `m5.2xlarge` on AWS, `n2d-standard-8` on GCP) |
| Load balancer | AWS NLB/ALB for ingress traffic |
| Persistent storage | EBS volumes for all stateful workloads |
| Refold application | All services deployed as Helm charts at the agreed version |
| TLS / certificates | Managed by cert-manager |

### Provisioning steps

<Steps>
  <Step title="Role validation">
    Pre-flight permission check — any missing permissions are reported before
    any resources are created.
  </Step>
  <Step title="Network bootstrap">
    VPC, subnets, NAT gateway, and route tables created in your chosen region.
  </Step>
  <Step title="EKS cluster + node groups">
    Cluster created in private mode; system and workload node groups
    configured.
  </Step>
  <Step title="Storage & secrets">
    EBS volumes and Persistent Volume Claims created; secrets provisioned
    inside the cluster.
  </Step>
  <Step title="Refold deployment">
    Helm charts deployed, health checks verified, smoke tests run.
  </Step>
  <Step title="Handover">
    Access and dashboard credentials are handed over to your team.
  </Step>
</Steps>

## Backups

Refold takes **hourly snapshots** of all stateful applications in your cluster.
Snapshots are stored as native EBS disk snapshots within your AWS account —
Refold never moves data outside your account boundary.

## Observability

A full observability stack is included with every BYOC installation. All
telemetry stays inside your account.

| Capability | Technology | Purpose |
|---|---|---|
| Metrics | Prometheus | Cluster, node, and application metrics |
| Dashboards | Grafana | Pre-built dashboards; credentials issued on handover |
| Logs | Loki + Promtail | Structured log aggregation and querying |
| Tracing | Jaeger | End-to-end request tracing |
| Alerting | Alertmanager | Routes alerts to Slack |

## Release management

Refold runs a control plane your DevOps team uses to manage upgrades. You
decide when upgrades run; Refold orchestrates them safely inside your cluster.

| Control plane (Refold-managed) | Data plane (your AWS account) |
|---|---|
| Version catalogue & release notes | EKS cluster and node groups |
| Upgrade orchestration | Refold application workloads |
| | Databases and stateful services |
| | All your data |

### Upgrade flow

<Steps>
  <Step title="Review">
    Log in to the control plane and check available versions and release notes.
  </Step>
  <Step title="Pre-flight check">
    The control plane validates cluster version and resource availability.
  </Step>
  <Step title="Rolling upgrade">
    Zero-downtime rolling upgrade; DB migrations applied with
    backward-compatible strategies.
  </Step>
  <Step title="Validation">
    Automated smoke tests run; health turns green once all checks pass.
  </Step>
</Steps>

| Upgrade type | Supported | Notes |
|---|---|---|
| Patch (3.1.0 → 3.1.2) | ✓ | Automated, no downtime |
| Minor (3.1.x → 3.2.x) | ✓ | May include DB migrations |
| Major (3.x → 4.x) | ✓ Guided | A Refold engineer coordinates a staged rollout |
| Rollback (same minor) | ✓ | Via the control plane, from the pre-upgrade snapshot |

## Roles & responsibilities

| Area | Refold | Client |
|---|---|---|
| IAM setup | Provides policy JSON and trust ARN | Creates the role, attaches policies, shares the ARN |
| Infrastructure | Provisions via the deployment role | — |
| Refold application | Deploys and patches | Triggers upgrades via the control plane |
