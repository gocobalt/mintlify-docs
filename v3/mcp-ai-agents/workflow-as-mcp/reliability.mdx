---
title: "Reliability"
description: "What workflows guarantee that agent-orchestrated API sequences cannot"
---

When an agent orchestrates a multi-step process by calling individual actions, every step is a decision the agent makes at runtime. The order, the retries, the rollback — all up to the agent. A workflow moves those decisions out of the agent and into a tested, deterministic procedure.

## What a workflow gives you

| Property | Agent-orchestrated actions | Workflow as MCP tool |
|----------|------------------------|----------------------|
| **Determinism** | Same input may produce different orderings across runs | Same input, same execution path every time |
| **Error handling** | Agent decides what to do on failure | Defined error paths per step |
| **Retries** | Agent decides whether to retry | Retry policy is set at workflow design time |
| **Rollback** | Not available | Compensating actions (reverse operations that undo prior successful steps) run on failure |
| **Business rules** | Enforced only if the agent remembers | Hard-coded in the workflow definition |
| **Auditability** | Each tool call audited individually | Single workflow execution audited with all steps |

## What "deterministic" means here

A workflow runs the same set of steps in the same order for the same input. The agent provides input and receives a final result; it has no influence over the intermediate steps.

This matters for processes where order is load-bearing:

- Approval gates that must run before the action they gate
- Validation that must succeed before any side-effect runs
- Compensating actions that must fire on partial failure

## Failure model

Each step in a workflow has an explicit failure path. When a step fails:

1. The workflow records the failure with the step name, the input it received, and the error from the downstream system
2. Any compensating actions defined for prior successful steps run
3. The workflow returns a structured error result to the agent (and to MCP Logs) — not a half-completed state

The agent receives one of: success with a structured result, a known business error (e.g., "vendor blocked"), or a system error. It does not need to reason about which step partially completed.

## Audit

Every workflow invocation is logged in [MCP Logs](/v3/mcp-ai-agents/security/mcp-logs) with the workflow ID, full input payload, final status, and duration. The individual step executions inside the workflow are recorded by the workflow engine itself and visible in the workflow execution history.
