MCP Logs is the audit trail for your MCP servers. Every tool call is recorded automatically — no configuration required. Use logs to debug failing actions, trace a full agent session, monitor integration adoption, and satisfy compliance reviews.

<Note>
For authentication and access control details, see [Authentication](/v3/mcp-ai-agents/authentication). For the MCP server architecture and request lifecycle, see [How it works](/v3/mcp-ai-agents/overview/architecture).
</Note>

## How MCP Logs work

Every request your MCP server handles writes one log entry before it returns. Logs are append-only. You cannot edit or delete individual records.

Key characteristics:

- **Retention** — records are kept for 90 days. Contact your account team for longer retention on enterprise plans.
- **Payload capture** — request inputs are captured up to 32 KB; response outputs up to 64 KB. Truncation is flagged in the record.
- **Downstream errors** — errors from third-party APIs (Salesforce, SAP, and others) are not surfaced verbatim to the agent. The agent receives a generic message; the full upstream error is stored in the `internal_error` field and visible in the detail panel.

Access logs in the Refold dashboard under **Maintain > MCP Logs**.

## What gets logged

Each log entry captures the following fields.

### Summary fields

| Field | Description |
|-------|-------------|
| **Status** | `success`, `pending`, `validation_error`, or `execution_error` |
| **Tool** | Which MCP tool was called (for example, `EXECUTE_ACTION`, `RESOLVE_ACTIONS`, `LOAD_SKILL`, or a direct-mode tool name) |
| **Action** | The specific action or workflow that was executed (for execution calls) |
| **Application** | Which connected app was involved |
| **Client** | Which MCP client made the call (for example, Claude Desktop, Cursor) |
| **Duration** | How long the call took, in milliseconds |
| **Timestamp** | When the call started |

### Detail panel

Click any log entry to open the detail panel.

<Frame>
  <img src="/images/mcp-logs-detail.png" alt="Audit detail panel showing request payload, metadata, and response" />
</Frame>

The detail panel shows four groups of fields:

**Request and response**
- Full JSON input the agent sent to the tool
- Full output returned by the tool (truncated at 64 KB for large responses)

**Identity**
- Linked account ID
- MCP server ID
- Organization ID
- Environment (`test` / `production`)

**Execution target** (for `EXECUTE_ACTION` and direct-mode tools)
- Application slug
- Action or workflow ID
- Target type (`action` or `workflow`)

**Client and transport metadata**
- MCP client name and version
- Protocol version
- Transport type (`streamable-http`)
- Session ID
- Trace ID (`X-Request-Id`)

## Filtering logs

Use the filter bar to narrow the log view.

| Filter | Use case |
|--------|----------|
| **Status** | Find all failed calls (`execution_error`, `validation_error`) |
| **Linked Account** | See all activity for a specific end user |
| **Application** | Filter to a specific app (for example, all Salesforce calls) |
| **Tool & Action** | Find calls to a specific tool or downstream action |

## Common use cases

**Debug a failing action** — filter by `execution_error` or `validation_error`. The detail panel shows the exact input payload the agent sent and the error returned. Start here when an end user reports that an agent couldn't complete a task.

**Trace a full agent session** — filter by Session ID to see every tool call from one agent session in chronological order. The Session ID is stable for the lifetime of a connection, so every `RESOLVE_ACTIONS`, `EXECUTE_ACTION`, `SEARCH_INTEGRATION_GRAPH`, and `LOAD_SKILL` call from one conversation is grouped together. You can see exactly what the agent did, in what order, and how long each step took.

**Monitor adoption** — filter by application or MCP server to see which integrations agents use most, how often they fail, and where response times are slow.

**Security review** — filter by linked account to audit what a specific user's agent accessed. Each log entry includes the full identity chain: organization, linked account, environment, and server.

## Compliance

MCP Logs provide the raw material for compliance and governance reviews:

- **Who** — every record ties to a specific linked account and organization
- **What was requested** — full input payloads captured up to 32 KB
- **What was returned** — full output payloads captured up to 64 KB; truncation is flagged
- **When** — timestamps with millisecond precision on start and completion
- **Duration** — milliseconds, for performance auditing
- **From where** — MCP client name, version, and transport metadata

For the broader security and data governance model, see [Security & compliance](/v3/mcp-ai-agents/implementation/security-compliance).

## Exporting logs

Logs are viewable and filterable in the dashboard. For bulk export or programmatic access, contact your account team.

## Rate limits

The MCP server does not impose per-tool rate limits. Plan-level limits apply to overall request volume. Rate limits enforced by a third-party application (for example, Salesforce API governor limits) propagate back to the agent as `execution_error` entries with the downstream error preserved in the detail panel.
