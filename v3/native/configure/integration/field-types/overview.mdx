A **field type** is the kind of input a [config field](/v3/native/configure/integration/config-fields) uses to collect a value from your customer. Every field you add to a connector has a type: a plain text box, a single-select dropdown, a mapping grid, a rule builder, and so on. The type decides what the customer sees, how they enter the value, and what shape the saved value takes.

Use this page to choose a type. For what config fields are, how customers fill them, and how to read and write their values from code, see [Config fields](/v3/native/configure/integration/config-fields). For the full palette and per-field settings in the platform, see [Dataslots](/v3/native/configure/integration/dataslots).

<Note>
In the platform, field types are added from a palette called **Dataslots**. The catalog below names each type and links to the page that explains it; the shared editor settings (Name, Required, Hidden, Default Value, and so on) live once on [Dataslots](/v3/native/configure/integration/dataslots#field-settings).
</Note>

## The catalog

Field types fall into three groups: **basic inputs** for fixed values you already know, **dynamic types** that pull live data from the customer's connected account, and **advanced types** that collect a mapping or a set of rules.

| Field type | Collects | When to use it |
| --- | --- | --- |
| **Text, Number, Email, Date** | A single typed value | A value you can't predict, like a prefix, a threshold, a recipient address, or a cutoff date. |
| **JSON** | A structured JSON value | A default payload or a list of values the customer edits as JSON. |
| **Select** | One (or several) choices from a list | A fixed set of options you type yourself, or live options when you back it with a [datasource](/v3/native/configure/integration/field-types/datasource). |
| **[Datasource](/v3/native/configure/integration/field-types/datasource)** | A choice from a list pulled live from the connected app | Account-specific lists that change per customer, like pipelines, stages, vendors, or owners. |
| **[Mapping (Map v2)](/v3/native/configure/integration/field-types/mapping)** | A mapping between your fields and the customer's fields | The customer lines up your data model against their connected app's properties. |
| **[Dependent](/v3/native/configure/integration/field-types/dependent)** | A choice that narrows based on another field | Cascading choices, where each selection filters the next (sheet, then worksheet, then headers). |
| **[Rule engine](/v3/native/configure/integration/field-types/rule-engine)** | A set of conditional rules | The customer builds if-this-then-that rules in your config UI. |

<Tip>
Reach for the simplest type that does the job. A static **Select** is clearer than a [datasource](/v3/native/configure/integration/field-types/datasource) when the options never change, and a **Select** is clearer than a [rule engine](/v3/native/configure/integration/field-types/rule-engine) when you need a single fixed choice rather than conditional logic.
</Tip>

## Basic inputs

**Text**, **Number**, **Email**, **Date**, and **JSON** are plain inputs with no live data behind them. They expose only the shared [field settings](/v3/native/configure/integration/dataslots#field-settings), and the **Default Value** input matches the type (a date picker for Date, a JSON editor for JSON, and so on). Use them whenever the value comes from the customer's head rather than from their connected account.

A **Select** sits between basic and dynamic: type a fixed list of options yourself, or back it with a [datasource](/v3/native/configure/integration/field-types/datasource) to load the list live. Turn on **Allow Multiple** for a multi-select.

## Dynamic and advanced types

The remaining types pull live data or collect more than one value at a time. Each has its own page:

- **[Datasource](/v3/native/configure/integration/field-types/datasource)** populates a Select's options from the customer's connected app, so each customer sees their own pipelines, stages, or vendors.
- **[Mapping (Map v2)](/v3/native/configure/integration/field-types/mapping)** lets the customer map your data model to their app's fields, then a workflow applies the mapping with the [Data Mapper node](/v3/platform/concepts/workflows/nodes/data-mapper).
- **[Dependent](/v3/native/configure/integration/field-types/dependent)** chains fields so a child field's options load only after its parent has a value.
- **[Rule engine](/v3/native/configure/integration/field-types/rule-engine)** gives the customer a grid to build conditional rules, with optional code to vary the operators and values per choice.
