A **mapping field** (the **Map v2** type) lets your customer match the fields in *your* product to the fields in *their* connected app. The customer sees two columns, your side and theirs, and lines up each value on one side with a value on the other. A workflow then applies the saved mapping to records, so the same connector syncs to every customer's field layout without bespoke code.

Use a mapping field whenever data has to move between your schema and a customer's, and the customer is the only one who knows how their fields are named. A classic case: matching your contact model (Name, Email, Phone) to the customer's HubSpot contact properties.

## How it works

A Map v2 field has two columns you configure independently:

- **Left-hand side (LHS)**: your side of the mapping, the fields or labels your product uses.
- **Right-hand side (RHS)**: the customer's side, the properties from their connected app.

You decide where each column's options come from. The customer maps each LHS value to an RHS value, and Refold saves the result as an object keyed by **LHS value → RHS value** under the field's `id`. Inside a workflow, the [Data Mapper node](/v3/platform/concepts/workflows/nodes/data-mapper) reads that mapping and applies it to records.

## Static vs. dynamic columns

Each column is either **static** (a fixed list you type) or **dynamic** (loaded live from the connected account). You can mix them: a static LHS against a dynamic RHS is the common shape.

| Column source | Configure it with | Use when |
| --- | --- | --- |
| **Static** | A typed list of options on the column | The values are fixed and the same for every customer, like your own data model's fields. |
| **Dynamic** | **Use Data Source**, or a [Dynamic Dataslot](/v3/native/configure/integration/dataslots#dynamic-dataslots) on the column's **Type** | The values are account-specific, like the customer's custom properties. See [Datasource fields](/v3/native/configure/integration/field-types/datasource). |

## Build a mapping field

<Steps>
  <Step title="Add a Map v2 field">
    In **Connect Portal → Fields**, add a field and set its type to **Map v2**, then edit it.
  </Step>
  <Step title="Configure the LHS column">
    Give the column a **Name** (its heading), set its **Type**, and supply its options. For a fixed list, choose **Select** and type the options. For account-specific values, check **Use Data Source** and pick a published [datasource workflow](/v3/native/configure/integration/field-types/datasource).
  </Step>
  <Step title="Configure the RHS column">
    Set the RHS column the same way. To map against the customer's app properties, choose a [Dynamic Dataslot](/v3/native/configure/integration/dataslots#dynamic-dataslots) for the column's **Type** (for example, *Map Contact Properties*), or back it with a datasource if no Dynamic Dataslot fits.
  </Step>
  <Step title="Save the field">
    Save the field. It becomes available on every linked account, and each customer fills in their own mapping.
  </Step>
</Steps>

<Note>
A column can also depend on another field. Set its **Dependency** so the column's options load only after the parent field has a value. See [Dependent fields](/v3/native/configure/integration/field-types/dependent).
</Note>

## Read and write a mapping from code

The saved value is an object keyed by **LHS value → RHS value**, stored under the field's `id`. Read it with `config()` and write it with `updateConfig()`, the same methods used for any field. See [Manage configs from your code](/v3/native/configure/integration/config-fields#manage-configs-from-your-code).

The config response lists the field's `options` (the RHS choices) and its `labels` (the LHS values you defined). Each label has a system-generated `value`, and you save the customer's mapping against those:

```json Config response (mapping field)
{
  "id": "65a0ee61b542ff4a62fe68b9",
  "name": "Map Deal Fields",
  "field_type": "map",
  "options": [
    { "name": "HubSpot Campaign", "value": "hs_campaign" },
    { "name": "Closed Won Date", "value": "hs_closed_won_date" }
  ],
  "labels": [
    { "name": "Name", "value": "65a0ef4db542ff4a62fe6a0c" },
    { "name": "Amount", "value": "65a0ef4db542ff4a62fe6a0d" }
  ]
}
```

```javascript Save the mapping
await refold.updateConfig({
  slug: "hubspot",
  config_id: "config_123",
  fields: {
    // field id -> { label value : chosen option value }
    "65a0ee61b542ff4a62fe68b9": {
      "65a0ef4db542ff4a62fe6a0c": "hs_campaign",
      "65a0ef4db542ff4a62fe6a0d": "hs_closed_won_date"
    }
  }
});
```

<Note>
For a static LHS, the label `value`s are system-generated and stable, so you map against them. For a dynamic LHS (one loaded from a datasource), the label values come from your datasource workflow instead. Pass dynamic labels via the `labels` argument on `config()`. See [Manage configs from your code](/v3/native/configure/integration/config-fields#manage-configs-from-your-code).
</Note>

## Best practices

<Tip>
- **Name your LHS labels for the customer**, in their language, since the labels are what they map against.
- **Use a Dynamic Dataslot for the RHS** where one exists, so the customer's real property names appear without you maintaining a list.
- **Apply the mapping with the [Data Mapper node](/v3/platform/concepts/workflows/nodes/data-mapper)** rather than reading the raw object yourself, it handles the transform per record.
- **Reach for [dependent fields](/v3/native/configure/integration/field-types/dependent)** when the RHS list depends on an earlier choice (an object, then its fields).
</Tip>
