A [linked account](/v3/platform/concepts/linked-account) represents one of your customers inside Refold. It is the scope that holds that customer's connections, credentials, and configuration, and you reference it by an ID you choose. This page covers the backend wiring: how to create a linked account and mint the [session token](/v3/api-reference/session-tokens/generate-session-token) your frontend needs to run the connect experience.

Your backend does two things per customer. First, create a linked account, typically when the customer signs up. Then, whenever that customer opens your integration UI, mint a fresh session token so the frontend can authenticate without ever seeing your API key.

<Note>
All calls use the base URL `https://app.refold.ai` and send your API key in the `x-api-key` header. Account-scoped calls also send the `linked_account_id`. Never expose your API key in the browser, mint a session token for the frontend instead.
</Note>

## How it works

The two objects play different roles, so keep them straight:

- **Linked account** is a durable record of one customer. You create it once and reuse its `linked_account_id` in every account-scoped call. Use a stable ID from your own system (for example, `user_12345`).
- **Session token** is a short-lived JWT scoped to one linked account. Your frontend sends it as `Authorization: Bearer <token>` so it can run the connect flow without your API key.

The flow is: your backend creates the linked account, your backend mints a session token, your frontend uses the token to drive the [Connect portal](/v3/native/frontend/overview) or React SDK.

## Create a linked account

Choose a unique `account_id` from your own data model and reuse it everywhere. Create the account when the customer first needs an integration.

```bash cURL
curl -X POST "https://app.refold.ai/api/v2/public/linked-account" \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "account_id": "user_12345",
    "name": "Acme Inc."
  }'
```

```json Output
{
  "_id": "6a3134fdc71d10070e78923e",
  "account_id": "user_12345",
  "name": "Acme Inc.",
  "environment": "test",
  "integrations": null
}
```

The `environment` in the response (`test` or `production`) reflects the API key you used. A linked account exists in only one environment.

<Tip>
You can seed customer metadata at creation with `udf` (user-defined fields) and pre-store credentials for your own app with `your_app.auth_credentials`. See [Create linked account](/v3/api-reference/linked-accounts/create-linked-account) for the full body.
</Tip>

<Note>
`account_id` must be unique within an environment. To create or update in one call, use [Upsert linked account](/v3/api-reference/linked-accounts/upsert-linked-account).
</Note>

## Mint a session token

When the customer opens your integration UI, mint a token for their linked account on your backend and pass it to the frontend.

```bash cURL
curl -X POST "https://app.refold.ai/api/v2/public/session-token" \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "linked_account_id": "user_12345" }'
```

```json Output
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_in": 599747,
  "refresh_expires_in": 2586947
}
```

By default Refold reuses an existing valid token. To force a brand-new one, pass `force_refresh=true` as a query parameter.

```bash cURL (force refresh)
curl -X POST "https://app.refold.ai/api/v2/public/session-token?force_refresh=true" \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "linked_account_id": "user_12345" }'
```

With the Node.js SDK, this is `getTokenForLinkedAccount`:

```javascript Node.js
const { token } = await client.getTokenForLinkedAccount({
  linked_account_id: "user_12345",
});
// Return `token` to your frontend.
```

<Warning>
A session token grants access to one customer's connections. Mint it server-side, return only the `token` to the frontend, and never log it or store it in client-accessible storage.
</Warning>

## Handle expiry

Session tokens are short-lived. Mint a fresh token for each new customer session rather than caching one indefinitely.

| Field | Meaning |
|-------|---------|
| `expires_in` | Seconds until the session token expires. |
| `refresh_token` | Use to obtain a new session token after expiry. |
| `refresh_expires_in` | Seconds until the refresh token itself expires. |

<Note>
Token TTL is set by service configuration, so read `expires_in` from the response rather than hardcoding a duration. If a request returns `401 Unauthorized`, mint a new token.
</Note>

## Common errors

| Response | Cause | Fix |
|----------|-------|-----|
| `400 INVALID_LINKED_ACCOUNT_ID` | The linked account does not exist in this environment. | Create it first, and confirm you are using the matching environment's API key. |
| `400` "linked_account_id is required" | Missing `linked_account_id` in the body. | Include it in the request body. |
| `401 Invalid Credentials` | Bad or wrong-environment API key. | Use the correct key from **Settings > Credentials**. |
