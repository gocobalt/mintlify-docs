title: "Data Import"
description: "Pull records out of a customer's connected app and load them into your product."

By the end of this recipe you will have records read out of a customer's connected app, reshaped into your own schema, and delivered to your backend, ready to run for every customer.

This recipe assumes you can already connect an app and execute an action (see the [Quickstart](/v3/native/quickstart)) and that you have built a [workflow](/v3/native/configure/integration/workflow) before. There are two ways to import, depending on volume.

## Two ways to import

Choose by how much data you move and how often:

- **Direct execute** suits a one-off pull or a small set of records. Your backend calls the connector's read action and handles the result inline. No workflow needed.
- **Workflow import** suits bulk loads, paging, per-record fetches, and scheduled refreshes. A workflow reads the records, reshapes them, and posts them to your server through an [API Proxy](/v3/connectors/my-app/api-proxies).

<Note>
Start with direct execute to prove the read works, then move to a workflow once you need paging, transformation, or a schedule.
</Note>

## Option A: Direct execute

For a small, on-demand import, call the read action straight from your backend with the customer's `linked_account_id`.

```bash cURL
curl -X POST https://app.refold.ai/api/v2/integration-schema/hubspot/actions/list_contacts/execute \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "linked_account_id: demo-user-1" \
  -H "Content-Type: application/json" \
  -d '{}'
```

```json Output
{
  "success": true,
  "data": {
    "results": [
      { "id": "1510721", "email": "jane@example.com", "firstname": "Jane", "lastname": "Doe" }
    ]
  }
}
```

Take the records from `data` and write them into your own store. Confirm the action name with the connector's [actions list](/v3/native/move-data/read-write) first.

<Warning>
`list_contacts` is a placeholder. Use the real read action for your connector from its actions list.
</Warning>

## Option B: Workflow import

For a bulk or scheduled import, build a workflow. It reads the records, transforms them, and delivers them to your server in one run.

<Steps>
  <Step title="Add the trigger">
    Create a workflow with an **event trigger** (fire it from your backend when you want a refresh) or a **scheduled trigger** (refresh on a cadence). See [Workflows](/v3/native/configure/integration/workflow).
  </Step>
  <Step title="Read the records">
    Add the connected app's node and select its read or list action. If the app returns IDs only, loop over them and fetch each record, collecting the results.
  </Step>
  <Step title="Reshape into your schema">
    Add a transform step to flatten and rename fields into the shape your backend expects. See [Data mapping](/v3/native/move-data/data-mapping).
  </Step>
  <Step title="Deliver to your server">
    Add an [API Proxy](/v3/connectors/my-app/api-proxies) node that POSTs the transformed records to your own endpoint.
  </Step>
</Steps>

### Run the import for a customer

Once the workflow is built and enabled for the linked account, fire its event to start a run.

```bash cURL
curl -X POST https://app.refold.ai/api/v2/public/event/hubspot \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "linked_account_id: demo-user-1" \
  -H "Content-Type: application/json" \
  -d '{
    "event": "Sync Contacts",
    "payload": {}
  }'
```

```json Output
{ "success": true }
```

<Note>
`Sync Contacts` is the event name you define on the workflow's trigger, and the slug in the path (`hubspot`) is the connector the workflow belongs to. A scheduled trigger needs no event call.
</Note>

<Check>
Your endpoint receives the transformed records, and they appear in your product. The same workflow now runs for every customer who connects the app.
</Check>

## Handle large imports

- **Page through results.** If the read action returns a cursor or page token, loop until there are no more pages.
- **Deliver in batches.** POST records to your server in chunks rather than one giant payload, so a single failure does not lose the whole run.
- **Make your endpoint idempotent.** Upsert by the connected app's record ID so a re-run does not create duplicates.
- **Watch the run.** Check workflow logs to confirm every page was read and delivered.
