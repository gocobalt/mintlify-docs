---
title: "Two-Way Data Sync"
description: "Keep records in sync in both directions between your app and a customer's connected app."
---

By the end of this example you will have records flowing both ways between your product and a customer's connected app: changes in your app push to the connected app, and changes in the connected app push back to you in near real time.

This example assumes you can already connect an app and execute an action (see the [Quickstart](/v3/native/quickstart)) and that you have built a [workflow](/v3/native/configure/integration/workflow) before. It reuses those building blocks and shows how to combine them into a sync. The example uses contacts; swap in any record type your connector supports.

## How two-way sync works

A two-way sync is three workflows working together, not one. Each handles a different direction or moment:

- **Initial sync** reads all existing records out of the connected app once, so both sides start from the same data. This is the [Data import example](/v3/native/recipes/data-import).
- **Outbound push** writes to the connected app whenever a record changes in your product. Your backend fires an [event](/v3/connectors/my-app/events); a workflow runs an action on the connected app.
- **Inbound push** writes to your product whenever a record changes in the connected app. The app's webhook trigger starts a workflow; the workflow posts the change to your server through an [API Proxy](/v3/connectors/my-app/api-proxies).

<Note>
Run the initial sync once per customer when they connect. Run the outbound and inbound workflows continuously after that. Keeping them separate is what makes the sync debuggable: each direction has its own logs.
</Note>

## Step 1: Seed both sides with an initial sync

Before anything streams, both sides must agree on the starting data. Build the import workflow from the [Data import example](/v3/native/recipes/data-import) and run it once for the linked account. After it finishes, your database holds every existing record from the connected app.

<Tip>
Store the connected app's record ID alongside your own record in this step. You will need that mapping to avoid creating duplicates when changes flow back later.
</Tip>

## Step 2: Push your changes out to the connected app

When a record changes in your product, send an event so a workflow writes the change to the connected app.

<Steps>
  <Step title="Build the outbound workflow">
    In the connector's [workflows](/v3/native/configure/integration/workflow), create a workflow with an **event trigger** and a node that runs the connected app's create-or-update action with the payload from the event.
  </Step>
  <Step title="Fire the event from your backend">
    When a user edits a record in your app, fire the event for that linked account.
  </Step>
</Steps>

```bash cURL
curl -X POST https://app.refold.ai/api/v2/public/event/salesforce \
  -H "x-api-key: $REFOLD_API_KEY" \
  -H "linked_account_id: demo-user-1" \
  -H "Content-Type: application/json" \
  -d '{
    "event": "Contact Updated",
    "payload": {
      "id": "internal-42",
      "first_name": "Jane",
      "last_name": "Doe",
      "email": "jane@example.com"
    }
  }'
```

```json Output
{ "success": true }
```

<Note>
`Contact Updated` is the event name you define on the workflow's trigger. The event slug in the path (`salesforce` above) is the connector the workflow belongs to.
</Note>

## Step 3: Push the connected app's changes back to you

When a record changes in the connected app, run a workflow that posts the change to your server.

<Steps>
  <Step title="Use the app's webhook as the trigger">
    Create a workflow whose trigger is the connected app's change event, for example a **Contact Created** or **Contact Updated** webhook. The workflow receives the changed record as its payload.
  </Step>
  <Step title="Post the change to your server">
    Add an [API Proxy](/v3/connectors/my-app/api-proxies) node that POSTs the payload to your own endpoint, so your backend can upsert the record.
  </Step>
  <Step title="Upsert by the stored ID">
    In your endpoint, look up the record by the connected app's ID you stored in Step 1 and update it, or create it if it is new.
  </Step>
</Steps>

<Warning>
Not every connector emits change webhooks. If yours does not, poll on a schedule instead: run the import workflow on a [scheduled trigger](/v3/native/configure/integration/workflow) and reconcile against your stored data.
</Warning>

## Step 4: Prevent sync loops

A naive two-way sync echoes its own writes: your outbound push triggers the app's webhook, which pushes the same change back, which can trigger another outbound push. Break the loop:

- **Compare before writing.** On each side, skip the write if the incoming value matches what you already have.
- **Track a last-modified timestamp** and ignore changes older than the one you last applied.
- **Tag writes you originate** where the connected app supports it, then skip inbound events that carry your tag.

<Check>
Edit a contact in your app and watch it update in the connected app. Edit a contact in the connected app and watch it update in yours. Confirm a single edit does not bounce back and forth.
</Check>
